# Architecture — rule-source-map (read rules from canonical locations, don't encode them here)

Rules live in their canonical place. This skill's job is to:
1. Read each canonical source.
2. Interpret what to check.
3. Run the check against actual repo state.
4. Report findings.

**Do not encode rule content in this skill body.** If a rule changes in its canonical source, the next `/repo-doctor` run picks it up. The skill stays correct without edits.

## Constraints

- **Read-only on existing files.** Only new file allowed: the audit report.
- **No network I/O.** Doctor checks structural state only. URL liveness is `/repo-update`'s job.
- **No environment changes.** Use what's already installed; degrade gracefully if `jq` / `python3` missing.
- **Stay in repo.** Verify via `git rev-parse --show-toplevel` before any path operation.
- **Banner at top and bottom of chat output.** Non-negotiable.
- **Stay out of `/repo-update`'s lane.** If you notice freshness drift (canon entry past `last_verified + cadence`), do NOT flag here — that's update's domain. Doctor checks broken-now, not stale-now.

## When the input is ambiguous

- `/repo-doctor` (no args) → run full diagnostic (all 5 tiers).
- `/repo-doctor <tier-keyword>` → match against `structural`, `schema`, `xref`, `format`, `invariants`; run only matching tier.
- `/repo-doctor help` → list tier keywords without running anything.

## How this skill evolves

When the repo's rule landscape changes:
- A new rule lands in a canonical source (CLAUDE.md, a README, the [research/sota-2026-v2/16-ai-readable-docs.md](research/sota-2026-v2/16-ai-readable-docs.md) authoring standard) → add a row to the rule-source map above.
- A term is retired → add a row to the audit-history table.
- A new structural invariant is documented → assign a check ID (D28, D30, ...) and add to the appropriate tier.

The skill body never encodes rule content directly — only the *map* of where rules live and how to check against them.
