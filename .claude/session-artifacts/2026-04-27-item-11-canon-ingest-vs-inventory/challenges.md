# Challenges — item 11: canon ingest vs. inventory

## Frame as given
The orchestrator reframed the punch-list's "reconcile two YAMLs" wording as a category error: the librarian greps [`canon/corpus/`](../../../canon/corpus/) directly and reads neither YAML, the two files are not parallel lists but a human-curated inventory plus a strict-subset fetch queue, and the residual decision is binary — close-with-corrected-rationale, optionally plus a lifecycle paragraph in [`canon/README.md`](../../../canon/README.md). Implicit optimization: don't paper over a wrong premise; correct the punch-list's rationale so the false alarm doesn't recur.

## Alternative frame
The session has been framed as a **documentation/closure** question (what to write, where, whether at all). An alternative frame is **provenance integrity**: the punch-list misread is a *symptom* that the canon subsystem has no single source of truth about *which file means what* — `sources.yaml`, `sources.ingest.yaml`, `corpus/`, the librarian's grep behavior, and `bin/ingest-canon.mjs`'s consumption pattern are each self-evident in isolation, but the *relationships* between them live nowhere. Under this frame, the verb is not "close" or "document the lifecycle" but **"locate the canonical statement of canon's own data flow."** The deliverable is not a paragraph in `canon/README.md` chosen by tradeoff against scope-creep; it is a one-page data-flow note (could be README, could be a comment header in `sources.yaml` itself, could be a docstring in `bin/ingest-canon.mjs`) whose *location* is chosen by where a confused future reader actually arrives. The current frame's binary collapses a placement question into a write/don't-write question and pre-decides the placement (README) without examining whether README is where misreaders land.

## Condition under which the current frame is wrong
The current frame is wrong if a second misread of the canon data flow is *already on disk* somewhere in the repo (an agent prompt, an upgrade entry, a workflow doc, a closed plan) and the orchestrator has not searched for it — because then the misread is no longer speculative, the outside-view's "lift condition" (a *second* attested misread) is already met, and "close as overtaken-by-events" is the wrong call.

## Challenges to scope-map preservations

- **`canon/sources.yaml` and `canon/sources.ingest.yaml` (both files unchanged, permanent two-tier index)**: preservation reason was "the asymmetry is permanent and correct because 10 owned-books have no fetch URL and belong in inventory only; the split is lifecycle-driven." That reason is weak in two specific places. (1) The owned-books-without-URL constraint argues only for *one bit per entry* ("fetchable: yes/no"), not for two physical files; a single `sources.yaml` with a `fetchable: true` field on each entry — and `bin/ingest-canon.mjs` filtering on that field at read time — preserves the same asymmetry with one fewer file, one fewer drift surface, and zero risk of `sources.ingest.yaml` having an entry that `sources.yaml` lacks (the exact failure mode commit 92d9387 just fixed). (2) "Lifecycle-driven" is doing rhetorical work that the artifact does not earn: the two files share a lifecycle (an entry is added to inventory and, if fetchable, simultaneously to the queue), so the split is not lifecycle-orthogonal — it is a denormalization for the convenience of the ingest script. Folding back into one file with a `fetchable` field would be the right move under any of the following conditions: (a) a third drift incident (commit 92d9387 was the second class of drift, after the original folder-shape mismatch); (b) the ingest script gains any additional per-entry filter (auth, rate-limit class, MIME), at which point the queue file becomes a partial mirror of inventory metadata and the denormalization stops paying for itself; (c) any future reader is observed treating `sources.ingest.yaml` as a separate authoritative list rather than a derived view (the punch-list author's misread is exactly this failure mode against the documentation, one step removed from the file). The preservation biases the downstream answer toward **"document the existing two-file shape"** when the more honest move may be **"replace the two-file shape with a one-file `fetchable:` field, after which there is nothing to document because the inventory and the queue are the same artifact."** Note that this is a frame challenge, not a recommendation: the generator may still choose preservation, but it must address why the denormalization earns its keep beyond "10 owned-books have no URL."

- **`canon-librarian.md` and `canon-refresher.md` (behavior preserved)**: preservation reason was "behavior is correct; the punch-list's claim that the librarian queries a YAML is the artifact that is wrong." That reason is weak because it conflates *correct behavior* with *legible behavior*. The punch-list author misread the librarian's query target; that is evidence the agent's query mechanism is *not* self-evident from its prompt. Preserving the agents without touching them means the next reader of [`canon-librarian.md`](../../../.claude/agents/canon-librarian.md) has the same chance of inferring "it must read sources.yaml because that's the inventory" — i.e., the same misread is reproducible from the same inputs. The preservation biases the answer toward **"the misread was a one-off reader error"** when an equally consistent reading is **"the agent prompt under-specifies its retrieval mechanism, and a one-line `Retrieval: greps canon/corpus/ directly; does not read sources.yaml or sources.ingest.yaml` would close the failure mode at its source."** Conditions under which touching the agent prompts is right: (a) any future agent invocation that produces output consistent with the YAML-query misreading; (b) a second human reader independently misreading; (c) an upgrade entry being drafted that presumes YAML-query semantics.

## Challenge to the classifier label
The classifier picked `investigation` with `extend` as alternative. The substantive risk in that pairing is that `investigation` framing collapses into "close the ticket" once the symptom is overtaken by events, while `extend` would force the orchestrator to ask "extend *what* primitive, and is that the right primitive to extend?" — which is exactly the placement question the alternative frame above raises. Under `extend` framing, the orchestrator would not default to README as the extension target; it would enumerate candidates ([`canon/README.md`](../../../canon/README.md), a header comment in `sources.yaml`, a docstring in `bin/ingest-canon.mjs`, the librarian's own prompt) and pick by reader-arrival probability. The classifier's choice of `investigation` as primary is defensible, but the orchestrator should notice that the moment the decision tilts toward "write something," the label has already flipped to `extend` and the placement question is live — not pre-decided as README.

## Frame-level objection the orchestrator should carry forward
Before recommending close-with-note vs. close-plus-README-paragraph, the generator must (a) name the conditions under which the two-file split itself should be folded back into a single `sources.yaml` with a `fetchable:` field — i.e., treat preservation of the two-tier index as a live design choice, not a settled premise — and (b) confirm whether a second misread of canon's data flow already exists on disk elsewhere in the repo, because that fact (not the README's silence) is what flips "speculative doc" to "attested doc."
