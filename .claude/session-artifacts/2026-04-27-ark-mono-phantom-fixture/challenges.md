# Frame challenges — ark-mono phantom fixture

## Frame as given

The orchestrator is holding three live frames in `frame.md`: F1 (punch-list closure — optimize for cost-to-close item 5), F2 (regression-discipline credibility — optimize for "exemplar exists by end of session"), F3 (triage — optimize for operator-attention preservation across items 6–13). The session is treated as an investigation of *which frame governs*, with the (a)/(b)/(c) options serving as the artifacts of each frame's choice. The orchestrator has refused to collapse to any one frame and is leaning on the parallel gather to produce data on which frame the repo culture actually prefers.

## Alternative frame

**F4 — calibration of the regression-test artifact itself.** All three live frames assume `tests/regression/ark-mono-connector-routing.md` is a *trustworthy specification* whose only problem is that it has never been run; the question becomes whether and when to run it. F4 inverts that: the unrun test is not a fixture-in-waiting, it is an *unverified hypothesis about what the agent stack should do*. Until it is read against the *current* agent definitions and CLAUDE.md (not executed — read), nobody knows whether its seven acceptance criteria still describe the system. Under F4, the unit of work is not "produce an exemplar," "patch docs," or "sequence attention" — it is **review the spec for currency**, treating the test like a stale ADR. The verb changes from *run/substitute/defer* to *audit*. After audit, the (a)/(b)/(c) menu reshuffles: if the spec is stale, (a) is incoherent (you'd be promoting output that matches a wrong target); if the spec is current, F2's argument strengthens; if the spec is partially stale, the correct first move is amending the test, which none of the current options name. The outside-view distillation's dominant failure mode ("authoring drift, neither defect nor test signal") is exactly the F4 case dressed in F1/F2/F3 clothes — the orchestrator is treating a calibration question as an execution question.

## Condition under which the current frame is wrong

The current frame (any of F1/F2/F3) is wrong if the seven acceptance criteria in `tests/regression/ark-mono-connector-routing.md` were authored against an earlier shape of the 12-step workflow (e.g., before the critic-panel split into three lenses, before the scope-mapper existed, or before the distiller step was inserted) — because then no option in the current menu produces a meaningful artifact: (a) produces a forced match or a forced mismatch against an obsolete target, (b) cements doc references to a test that should be amended, and (c) defers a question that will recur in identical shape until the spec is reconciled with the current workflow.

## Challenges to scope-map preservations

- **Session-artifacts immutability norm**: preservation reason was "all three options are forward-only; (a) writes new artifacts and a new exemplar copy; (b)/(c) edit only doc citations, not historical artifacts; preservation is the user's stated norm." That reason is weak because option (a)'s "promotion to exemplar" step is *not unambiguously forward-only* — it copies (or hard-links, or renames-with-history) a session-artifact directory into `.claude/session-artifacts/exemplars/` under a new canonical name. The original session-id directory is preserved on disk, but the *promoted* artifact is a derivative whose existence depends on a session that has been (a) curated, (b) possibly edited for exemplariness, (c) re-stamped with an exemplar-shaped slug. Three sub-challenges: (1) does promotion permit *any* edit between session and exemplar? If yes, the exemplar is a soft rewrite under a new filename, and the immutability norm has been violated in spirit even though the source dir remains. If no, the exemplar inherits every wart of a real session, including the "first execution" warts the outside-view forecast says are likely (~80% probability of revision per its estimate). Either branch undermines the preservation. (2) The norm protects *historical record*; an exemplar is a *prescriptive artifact* (others will be authored to match its shape). Preserving the immutability norm while creating a prescriptive copy means the prescriptive artifact inherits historical-record protection it does not deserve — future drift in the workflow cannot edit the exemplar without violating the norm. (3) The scope-map's claim that (b)/(c) "edit only doc citations, not historical artifacts" is itself questionable: CLAUDE.md and the `session-artifacts/README.md` are not historical session-artifacts, but they *cite* a slug that will, under (a), become the name of an exemplar — i.e., the doc edits in (b)/(c) are being judged immutability-safe by a rule that doesn't apply to docs in the first place, which means the preservation is doing no work for two of the three options and is doing *contested* work for the third. The preservation biases the downstream answer toward **option (a) appearing norm-compliant when it is the option most likely to require a post-promotion edit, and toward (b)/(c) appearing norm-neutral when the norm was never the constraint they were trading against.**

## Challenge to the classifier label

The classifier landed on `investigation` with a substantive alternative of `new` (option (a) is functionally the creation of the first real exemplar fixture). The alternative is substantive enough to flag: under `new`, the downstream frame would no longer be "which frame governs?" but "given that we are building the first instance of a prescriptive artifact class, what are the construction standards?" — and the conversation would shift from (a)/(b)/(c)-picking to *what makes an exemplar canonical* (does it need a pre-committed acceptance contract? a review gate? a versioning scheme tied to workflow revisions?). The outside-view forecast's "what would lift option (a) above base rate" list (pre-committing expected outputs, a wrong-spec-vs-wrong-agent rule, a max-edits cap) is exactly the construction-standards conversation the `new` framing would force. Staying with `investigation` lets the orchestrator treat the construction question as out-of-scope; switching to `new` makes it the *whole* scope. The frame bias the classifier flagged ("ritualizing the choice without forcing the frame question") may itself be a symptom of the wrong label — investigation invites frame-debate, new would invite construction-standards debate, and the latter is the conversation the canon-librarian's stub-shaped gap (Beck/Feathers/Meszaros on characterization tests and golden-master hygiene) is begging for.

## Frame-level objection the orchestrator should carry forward

The generator must address: *the menu (a)/(b)/(c) presupposes the regression-test spec is current; if the spec was authored against an earlier workflow shape, every option in the menu is the wrong unit of work, and any recommendation that does not include a spec-currency audit before option selection is generating against an unverified premise.*
