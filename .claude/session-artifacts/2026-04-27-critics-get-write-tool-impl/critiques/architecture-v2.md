# Architecture critic — verdict for `2026-04-27-critics-get-write-tool-impl` (loop 2)

## 1. Weakest structural link

**v2-C's "byte-fidelity prose contract on the orchestrator" relocates the contract from the place it could be enforced by construction (a typed return shape) to the place it cannot be enforced at all (orchestrator self-discipline mid-context).** In v2-A, "verbatim" is mechanically true: the orchestrator copies a string field from a structured return and the operation either succeeds or fails as a pure data move. In v2-C, "verbatim between delimiters" is a behavioral claim about an autoregressive sampler that is simultaneously holding the critic's prose, the contract text, and a live conversation context in one window — and is asked to reproduce the prose without "improving" it. That class of instruction is the *exact* class LLMs are weakest at: cross-turn high-fidelity transcription of prose under attention pressure. The candidate names this in A1 and "Ways v2-C could be wrong #1," then argues detection-after-the-fact is sufficient because the operator can diff the orchestrator's transcript against the file. Detection-after-the-fact is not enforcement; it is monitoring. The structural defect is that v2-C *picks the weaker mechanism out of the two it itself enumerates* (prose contract vs. structured return) and justifies the choice by parsimony. Parsimony of primitives is not parsimony of guarantees.

## 2. Invariants at risk

1. **Delimiter-injection resistance.** The contract says "copy the bytes between `<critique>` and `</critique>` verbatim." A critic whose prose body legitimately contains the substring `</critique>` (e.g., a critic *quoting another critic's delimiter usage in a future loop*, or this verdict you are reading right now, which discusses delimiters) silently truncates. The candidate ships no escaping rule, no "the closing delimiter must be on its own line and the body must not contain that line" invariant, no length-prefix alternative. v2-A's structured return has no such failure mode because field boundaries are not in-band.

2. **Round-trip identity under whitespace and unicode normalization.** "Verbatim" is undefined for: trailing whitespace the orchestrator's tooling strips, NFC/NFD unicode reordering, CR/LF normalization, tab-vs-space conversions performed by an editor or by the Write tool itself. A "byte-fidelity" contract that does not specify the encoding contract is byte-fidelity in name only. Today's paraphrase failure mode is at least *visible*; v2-C's normalization failure mode is invisible by construction (the operator's diff tool will also normalize).

3. **Operator-surface coherence across sessions.** v2-C drops the aggregate. Step 12 reads three per-lens files. The invariant *the operator can read the panel's verdict from a single named artifact* is broken. The candidate frames this as "one CLAUDE.md sentence's worth of retraining" — that is a habit-cost framing of a *structural* property: the panel as a single object with a single verdict-surface no longer exists. If a fourth lens is added later (this is plausible — `critic-security` is the obvious one), there is no schema to extend; there is a directory that grows. v2-A's structured return forces a schema decision now; v2-C defers it forever, which means it never gets made.

4. **Convergent verdict readability.** Today's `critiques.md` has one place where "all three verdicts" can be read in one glance. v2-C's three files require three reads. For minority-veto routing (Step 10), the orchestrator must hold three open files in context to compute the verdict — a regression from "open one file, scan three labels." Step 12 is named; Step 10's read pattern is not, and it is the load-bearing one.

## 3. Coupling and direction

- **Contract direction inverted vs. v2-A.** In v2-A, the *return shape* (data) carries the contract; the orchestrator is a dumb writer. In v2-C, the *orchestrator's prose-following behavior* carries the contract; the critic is a dumb prose-emitter. v2-C couples the contract to the most-volatile component (the orchestrator's per-turn instruction-following), where v2-A couples it to the most-stable (a typed schema). This is a stable-depends-on-volatile inversion the candidate frames as "preserve today's chokepoint" — it preserves the chokepoint's *location*, not its *strength*.
- **Delimiter convention is now load-bearing across three agent files plus CLAUDE.md.** Four files share a string literal (`<critique>`). The candidate proposes no central definition. Drift across the four (one critic updates its template, the orchestrator's contract still references the old delimiter) is an undetectable failure: the orchestrator copies whatever-is-between-whatever, and the file looks fine. v2-A makes this impossible because a typed field name lives in code, not in four prose copies.
- **Mesh-via-filesystem objection from loop 1 is resolved.** Critics no longer write; sibling-read is structurally impossible. Star topology is preserved. This is a real win and I credit it.
- **Naming-implementation drift from loop 1's Defense 2 is resolved by removal.** Also a real win.

## 4. Ignored architectural alternatives

1. **v2-A with explicit per-section attribution metadata.** The candidate dismisses v2-A on the grounds that "v2-A and v2-B do not propose attribution metadata, so they are equivalent to v2-C on attribution." That is a dismissal of v2-A-as-currently-sketched, not of v2-A-as-architecturally-possible. v2-A with a `{verdict, body, lens, session_id, agent_version}` shape carries provenance *in the artifact itself* — superior to v2-C on every audit property without losing any v2-C property except sentence-count in CLAUDE.md.

2. **Hybrid: structured return + per-lens file + retain a generated aggregate.** Critics return typed shape; orchestrator writes per-lens files (operator surface, as v2-C wants) *and* generates an aggregate index by mechanical concatenation of the typed `verdict` field across lenses. Cost: same as v2-A. Benefit: preserves Step 10's one-glance read, preserves Step 12's deep read, eliminates v2-C's invariant-3 regression. The candidate dropped the aggregate as a binary choice (keep-as-thin-index vs. drop-entirely) and never considered "aggregate as deterministic projection of the per-lens files."

3. **Content-addressed verbatim envelope.** Critics return prose; orchestrator computes a hash of the prose, writes both the prose and the hash to the file as a header (`# sha256: <digest>`). Detection of paraphrase-despite-contract becomes mechanical (recompute, compare) rather than requiring the operator to manually diff against a transcript. Costs one orchestrator-side hash call. v2-C's A1 detection story becomes structural rather than behavioral.

## 5. Frame-level objection

The candidate's loop-2 self-correction ("Revision 2's frame may be over-fitted to architecture's vocabulary") is *partially fair and partially face-saving*. Fair: the orchestrator did over-read "convergence" in loop 1. Face-saving: the candidate uses that admission to demote the parliament frame *while keeping its strongest structural conclusion* (single-writer chokepoint preserved) and rebranding the conclusion as "per-lens objection-resolution" rather than "frame adoption." The honest move is to say: *the parliament frame was correct on the role-separation point, and v2-C inherits that correctness; what was wrong was treating the parliament frame as the only frame the loop-1 verdicts implied.*

The genuinely fresh frame-level objection v2-C exposes that was not visible in loop 1: **byte-fidelity is not a single property — it is a stack of properties (encoding, escaping, normalization, transcription, attribution), and the candidate addresses one (transcription) while leaving the other four undefined.** A change that pays a CLAUDE.md-sentence cost to deliver "byte-fidelity" without specifying which layer of the stack it operates at is a change that ships an undefined guarantee. v2-A's structured return forces three of the four layers to be defined by the schema; v2-C defers all four to "the orchestrator follows the contract."

## 6. Verdict

**rework.**

What would change the verdict: switch to v2-A (or the hybrid in alternative #2), or — if v2-C is retained — add (a) an explicit delimiter-escaping rule, (b) an encoding/normalization specification for "verbatim," (c) a content-addressed envelope (alternative #3) so paraphrase-despite-contract is mechanically detectable rather than transcript-diffable, and (d) a named plan for Step 10's three-file read pattern. v2-C is closer to right than the loop-1 candidate — role-fusion is genuinely resolved, mesh-coupling is resolved, Defense-2 drift is resolved — but it picks the weaker of the two enforcement mechanisms it itself names and ships an undefined byte-fidelity guarantee.
