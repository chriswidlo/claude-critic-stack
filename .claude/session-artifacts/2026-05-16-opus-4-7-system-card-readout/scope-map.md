Two findings (F3, F5) produced no rows because the implicated primitive class does not exist in this repo.

The dominance of `extend` over `subsume`/`replace` is itself a finding the generator must address rather than absorb silently. Three possible readings:

1. **The repo's primitives are well-scoped against vendor-document-readout work.** The findings update *parameters* (named failure modes, additional anchor-risk classes, sharpened anti-anchoring wording), not *contracts*. If true, this is a positive signal about prior architecture.
2. **The mapper is defaulting to `extend` because the findings do not in fact bear on agent contracts.** The classifier flagged "research theater" as the dominant frame bias; an all-extend column is consistent with that bias materializing — findings landed but did not move primitives. The generator should test this by asking, for each `extend` row, whether the extension would actually change a future workflow run's behavior, or whether it is documentation that adds no observable change.
3. **The honest answer for several rows is `replace`, not `extend`, and the mapper softened.** Candidates: F1's effect on the shadow-lane rationale arguably *replaces* the prior justification for SHADOW_PANEL (currently framed as "correlated-review error," would be reframed as "audit-context behavior divergence"). F4's effect on outside-view arguably *replaces* the agent's failure-mode taxonomy rather than extending it. The frame-challenger step should test whether `extend` here is the non-committal answer the scope-mapper contract warns against.

The single `conflict` (F2 × Opus critic lenses) is the load-bearing row. It cannot be resolved by the mapper because the repo has not chosen between accepting elevated refusal as workflow cost, routing safety-flavored candidates to shadows, or restructuring critic prompts. This goes to the user, not to the generator.

## Requires decision (conflicts)

- **F2 vs. critic-architecture / critic-operations / critic-product (Opus-pinned lenses):** A 12% → 33% refusal regression on AI-safety R&D tasks means the three Opus critic lenses may refuse to evaluate exactly the safety-flavored candidate recommendations this stack produces. Both cannot coexist as currently configured because (a) the lenses are required to return a verdict for the minority-veto rule to be sound, and (b) a refused return is structurally indistinguishable from `unavailable`, which the comparator already treats as a missing-data case. User has not chosen between: (i) accepting elevated refusal as a workflow cost and tolerating veto-stopped sessions, (ii) routing safety-flavored candidates to Sonnet shadows as primary (which subverts the "shadow has voice, not vote" rule), (iii) restructuring critic prompts to avoid the refusal trigger (which risks gaming the regression rather than measuring it).
