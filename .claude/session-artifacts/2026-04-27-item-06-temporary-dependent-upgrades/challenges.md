# Frame challenges — item 06 temporary-dependent upgrades

## Frame as given

The orchestrator's frame is dual-track: (1) the user's local-consistency frame — pick whichever of (a)/(b)/(c) most accurately represents preservation of the underlying ideas while making `main` reader-honest; (2) the alternative reframe naming option (d), the "missing dependency-contract rule." Both frames assume the *entries themselves* are objects worth resolving — that something must happen to them on `main`, and the only question is which state-token plus body-text combination is most truthful. The orchestrator is staged to pick from a four-way option set whose elements are commensurable along one axis (how the entry's row reads).

## Alternative frame

**This is a graduation question, not a state-token question. The right primitive to interrogate is not the entry's row — it is the entry's *right to be on `main` at all*.** Re-verb the question from "how do we represent these entries' state" to "do these entries pass the bar for living in the canonical tree, given that they cannot be executed against the canonical tree?" Under this frame, the lab has implicitly run a CI check ("can this entry be acted on against `main`?") and the two entries failed it. The honest output of a failed CI check is not a state token — it is **eviction**: move both entries off `main` (delete from the LEDGER, delete the directories, let them live in `agentic-engineering-research` where their dependency *does* resolve, and let them re-enter `main` only if and when the worktree branch graduates). The optimization target flips from "represent preservation accurately on `main`" to "keep `main` populated only by entries whose dependencies are in scope for `main`." The four-way option set collapses: (a) is "promote dependency to make entries valid," (b)/(c) are "downgrade entry-state to mask the validity gap," (d) is "write a rule about the gap" — all four assume the entries should remain in the canonical tree. The graduation frame says they should not, until their dependency does.

This is a frame, not a recommendation: it does not name a mechanism for eviction, what gets back-referenced from the LEDGER, or whether the parent ideas are captured elsewhere first. It only re-verbs the question.

## Condition under which the current frame is wrong

The current frame is wrong if `main`'s LEDGER is contractually a *canonical, executable* index (every row must be actionable from `main`'s tree alone), rather than a *reflective, narrative* index (rows may record ideas whose execution lives elsewhere) — because in the canonical-executable reading, both entries are violating an invariant and the only correct response is removal, not retitling.

## Challenges to scope-map preservations

- **`agentic-engineering-research` worktree branch (preserved).** Stated reason: the plan's §Out-of-scope clause says "preserved as the source of truth; touching it is a different decision." That reason is weak because *the user wrote the §Out-of-scope clause in the same plan that created the dependency confusion the workflow is now trying to clean up* — the constraint is self-imposed by the same operator whose unresolved choice (backup vs archive, per the scope-map's own conflict row) is the root cause. Treating the branch as untouchable is fine as a sequencing rule ("we will not edit it during item 6"), but the scope-map has imported it as a structural primitive ("the branch's content is not modified by any of the four options"), which silently rules out a fifth option: **fold the worktree branch back into `main` under a clearly demarcated `research/` or `archive/` subtree, then both entries' dependencies resolve trivially and the parallel-branch problem disappears**. The preservation biases the downstream answer toward solutions that *accept* the cross-branch split as permanent geography, when the cleanest fix may be to eliminate the geography. It also smuggles in operator reluctance ("I don't want to delete the work I did on that branch") as if it were a technical invariant; reluctance to delete is not the same as a constraint that the branch must remain a separate branch.

- **The two affected entries' parent ideas (preserved).** Stated reason: "the *idea* survives even under option (b) — only the executable migration plan retires; concrete cost of deletion is losing the captured-at-`🌱` thinking that produced both entries on 2026-04-26, which is non-recoverable from git history alone." That reason is weak because *non-recoverability from git history is the standard cost of any deletion* — the lab's own honest-negation doctrine asserts that some ideas should be killed and named-as-dead rather than kept alive via state-token gymnastics, and the preservation argument here is identical to the argument for never killing any 🌱 entry ("the thinking would be lost"). The preservation also conflates two things the scope-map separated elsewhere: the *parent shape narrative* (durable module for power-user knowledge; reference-library shape) and the *specific 2026-04-26 capture of that narrative as an upgrade entry*. The parent shapes are abstract and survive in the operator's head, in CLAUDE.md-style guidance, or in a future entry; the specific entry-bodies are the artifact actually under question. Preserving "the idea" is cheap rhetoric that lets the workflow avoid asking whether *these specific entry-bodies, written when the operator believed `temporary/` would migrate to `main`*, still represent the operator's current thinking now that that belief has been falsified. The preservation biases toward (b)/(c)/(d) (which keep the entries) and against any frame in which the right move is "delete the entries; re-create one or both later from a clean prior if and when the dependency lands" — i.e., it pre-commits the workflow against the graduation frame above.
