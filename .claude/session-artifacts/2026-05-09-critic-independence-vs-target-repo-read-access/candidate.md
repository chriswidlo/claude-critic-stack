# Candidate recommendation (revision 2 — post-rework)

Revision 1 was rework'd unanimously by all three lenses (verdicts at [critiques.md](.claude/session-artifacts/2026-05-09-critic-independence-vs-target-repo-read-access/critiques.md) and per-lens files under [critiques/](.claude/session-artifacts/2026-05-09-critic-independence-vs-target-repo-read-access/critiques/)). Architecture flagged "claim change before mechanism change"; operations flagged "no detectors, all moves are prose"; product flagged "marketing downgrade without product-level care." This revision picks a different position from the panel's three named alternatives (claims.md change with veto removal, mechanism-introduction, or claim-only with no veto) and is deliberately **smaller and deferred**: it answers the user's question as a diagnosis with named follow-up artifacts rather than a CLAUDE.md edit ready to ship.

## Position

**The user's question is answered as a diagnosis, not as a contract change.** Three things are true given the evidence:

1. **No "introduction" of critic target-repo read access is happening.** Critics already hold unconstrained `Read`, `WebFetch`, `WebSearch` (Explore §1–11). The thing being introduced is *formalized routing* via `critique-prep` + `critique-start` + `$CRITIC_TARGET`. Influence on the panel today is shallow but unmeasured: no observed session has exercised critic target-repo access; the dominant correlation source is candidate.md authorship by the orchestrator after Explore.

2. **The independence claim was never information-independence.** The stack has *role-orthogonality* on shared input (architecture / operations / product prompts) and *optional inter-model triangulation* under `SHADOW_PANEL=1`. It has never had, and was never structured to have, information independence between lenses. Reading "three independent critics" as "three lenses with separate informational priors" was always inference, not contract.

3. **A panel-wide contract change is premature.** Outside-view places panel-wide isolation below base rate. Canon contradicts itself: SRE PRR / code review favor grounding for holistic critique; Anthropic multi-agent + forensic blinding favor input separation for narrow tasks. The dominant correlation source (candidate.md framing) is upstream of any panel mechanism, so any new isolation primitive is operating downstream of the binding constraint until that source is measured.

**Recommendation:** ship nothing structural. Open three named follow-up artifacts that capture what each option would cost; run the cheap probe in one of them; let the result determine which option, if any, to take.

## Named tradeoffs

- **Velocity vs. substantiated changes.** This session ends without contract changes shipping. The user came with a decision request and leaves with a diagnosis + three named follow-up artifacts. Trade: the user must wait for the probe to run before the structural picture changes; in exchange, no load-bearing contract is amended on the basis of unmeasured correlation.
- **Discipline cost vs. measurement-first.** Move 2 (probe upgrade) requires authoring a small, deliberately bounded experiment; move 3 (mechanism upgrade) requires designing three architectural alternatives without choosing. The cost of authoring three upgrade entries is real but additive, not load-bearing — the lab is creative-ideas hub per memory, and an unread entry is cheaper than a vetoed contract change.
- **Documentation accuracy vs. claim continuity.** Move 4 (footnote) shifts the panel's stated property from implicit "independent" to explicit "role-decorrelated on shared input." This is smaller than revision 1's CLAUDE.md rewrite but still a public commitment shift — softer than retraction, harder than silence. Trade: a small honest claim now beats a larger claim awaiting future violation.

## Named assumptions (any one wrong flips the recommendation)

- **A1 — The probe is informative at N=1.** Revision 2 keeps move 4 (the experiment) but architecture's idempotence objection holds: a single session of paraphrased-vs-original candidate under shadow may not generalize. If A1 is wrong, the probe upgrade entry must specify a multi-session protocol (which inflates the experiment's cost and pushes it back into "design properly before running") — at which point the diagnosis-first synthesis remains correct but the probe is no longer cheap.
- **A2 — Three named follow-up artifacts is the right granularity for deferred work.** This recommendation rests on `upgrades/` being the right destination for designed-but-not-chosen alternatives (per `feedback_lab_is_creative_hub` memory: lab entries are allowed to be wrong; cleanup tasks must not edit entry bodies). If A2 is wrong (e.g., the user reads an upgrade entry as a commitment rather than a sketch), the deferred-decision framing collapses and revision 2 needs to commit to one of the three architectural options now.
- **A3 — A footnote is sufficient documentation accuracy.** Move 4 names the panel's actual property in a footnote rather than rewriting the panel's headline description. If A3 is wrong (a footnote is read as hedging rather than clarification), product's "marketing change masquerading as a clarification" objection re-surfaces against the smaller move. Mitigation: move 4 is structurally additive (no existing claim is retracted), so even under A3 failure, the worst case is "user notices a footnote and reads CLAUDE.md more carefully" — not a marketing round-trip.

## Ways this could be wrong

- **W1 — Diagnosis-first is a dodge.** The user asked a decision question ("how should we approach it"); answering with three upgrade entries and a footnote can read as workflow theater that produced a lot of artifacts and no decision. Mitigation in synthesis: name the deferred decisions explicitly, name what would make them re-openable, name the cheapest experiment that would resolve the biggest one.
- **W2 — Three upgrade entries is two too many.** The lab entries are cheap but the act of *authoring* three of them in this session inflates session cost. The ledger will reflect this. If the user reads ledger:agent-calls and concludes the workflow over-spent, revision 2 has produced a defensible answer at a cost they would not have paid voluntarily.
- **W3 — The probe upgrade entry contains the same architectural defect that revision 1 contained.** The probe (paraphrase vs. original candidate under shadow comparator) inverts the comparator's documented semantics — architecture's §3 objection. Revision 2 addresses this by stating the inversion in the upgrade body and asking the comparator's protocol surface be updated *before* the probe runs, but if that update never happens, the probe runs against an undocumented contract and produces a measurement against an unstated baseline.
