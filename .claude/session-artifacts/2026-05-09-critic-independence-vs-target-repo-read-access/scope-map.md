## Requires decision (conflicts)

No primitive in the enumerated set is in *technical* conflict with the new requirement — every relationship resolves to extend or (for one clause) replace. However, the frame surfaces a **frame-level conflict** that the generator must address but that the scope-mapper does not pre-decide:

- **Panel-wide information-isolation regime vs. existing role-orthogonality independence claim.** If the user, in step 9, chooses option (5) from frame.md ("accept that critics are grounded; drop the information-independence claim; re-justify the panel on role-orthogonality + shadow-comparator"), then the critic-panel primitive's relationship flips from `extend` to `replace` — the *justification* of the panel changes even if the lens triple does not. The user has not chosen between (a) per-lens job declaration (extend) and (b) re-justification (replace). Frame-challenger and the generator must force this choice; the scope-mapper records the dependency.

- **Forensic-blinding reference class vs. LLM-as-judge reference class** ([outside-view.md §Contradictions](.claude/session-artifacts/2026-05-09-critic-independence-vs-target-repo-read-access/distillations/outside-view.md)). Whether *any* lens should be informationally isolated turns on whether that lens's job is "narrow technical conformance" or "holistic judgment." Outside-view named `critic-operations` as the only plausible isolation candidate; canon's PRR passage ([canon.md §6](.claude/session-artifacts/2026-05-09-critic-independence-vs-target-repo-read-access/distillations/canon.md)) directly contradicts that — operations-readiness review is *defined by* grounding. User has not chosen the reference class. Generator must pick or hold.

## Preserved primitives with stated reason (non-default)

- **Three-lens critic panel structure (architecture / operations / product)**: preserved because canon's role-orthogonality argument ([canon.md §inferred 4, 6](.claude/session-artifacts/2026-05-09-critic-independence-vs-target-repo-read-access/distillations/canon.md)) plus outside-view's "independence is operationalized relationally/structurally, not informationally" finding ([outside-view.md §inferred 2](.claude/session-artifacts/2026-05-09-critic-independence-vs-target-repo-read-access/distillations/outside-view.md)) jointly establish that the lens triple is the *correct* independence mechanism for holistic review. Removing it to reach for input-isolation would trade a primitive with empirical support for one without (zero-of-four reference classes, per outside-view).
- **Shadow comparator and `SHADOW_PANEL=1` gating**: preserved because it addresses a different correlation source (inter-model) than the one this question targets (inter-input). Replacing it would surrender cross-family triangulation to gain nothing on the input axis.
- **Path-discipline regime**: preserved because it is non-negotiable per [CLAUDE.md](CLAUDE.md); user has stated explicitly that the design-question repo and target codebase separation is load-bearing. The new requirement layers on top.

## Direct answer to the user's framing question

**The new requirement extends the existing minority-veto independence model; it does not replace it.** Replacement would mean removing the minority-veto rule or the three-lens structure, which the canon and outside-view evidence both argue against. The requirement *adds* an inter-input independence concern that the current stack does not address — concretely: a per-lens declaration of whether the lens is a calibrated (grounded) judge or an abstract (ungrounded) judge, and an explicit pass-through contract at step 10 that replaces the ambiguous "if they verify facts" clause. Minority-veto continues to govern verdict aggregation; the addition governs *what the verdict is a verdict about*.

The one local replacement is the conditional clause in [`critique-start`](.claude/skills/critique-start/SKILL.md) lines 38–42, which becomes an explicit per-lens table.
