# Frame challenges — karpathy-skills adoption (meta-validation run)

## Frame as given

The orchestrator framed the question as **regret-minimization across a small set of plausible actions** (do nothing / drop a line / install plugin globally / install plugin per-repo / fork the packaging pattern), ranked by recovery cost rather than by perceived correctness. The "extend" classifier label is retained — the unit of action is a tweak to existing primitives — but the decision criterion is shifted from "is the prior writeup optimal?" to "which action's failure mode is cheapest to undo?" Frame B (ecosystem-signal: treat this as the first of N future plugins) is held as a sanity check on the chosen action. The generator is told to enumerate at least four actions and name the upside that would justify each more-expensive action's recovery cost.

## Alternative frame

Reframe as **"the prior recommendation is already an artifact of the same workflow that is now being asked to validate it; the right question is not which action to take but whether the workflow is capable of returning a verdict other than 'the prior recommendation stands, perhaps with cosmetic tweaks.'"** The verb changes from *choose-an-action* to *audit-the-procedure*. Under regret-minimization, the cheapest action almost mechanically wins (do-nothing and drop-a-line both have ~zero recovery cost; everything else is dominated by definition), which means the frame is structurally rigged to confirm the prior recommendation. A frame that took the meta-validation seriously would instead ask: what evidence, if surfaced by the workflow, would cause the orchestrator to recommend an action *more expensive* than the prior one — and is the workflow set up to surface that evidence, or only to surface confirmations of low-cost defaults? This is not an alternative action; it is an alternative *target of optimization* (procedure integrity rather than action selection). It changes the downstream answer because the generator step is now obliged to name the falsifier-of-the-prior, not just to rank actions.

## Condition under which the current frame is wrong

The regret-minimization frame is wrong if recovery cost is not the binding constraint — specifically, if the user is paying a continuous *opportunity cost* from the empty global CLAUDE.md (every coding session that proceeds without test-first / clarify-first standing instructions is a session where the gap silently degrades output), in which case the relevant axis is **expected value of the always-on lift over the next N sessions**, not recovery cost of a future undo. Under that condition, "do nothing" stops being a near-zero-cost action and becomes a slow bleed, and the ranking inverts.

## Challenges to scope-map preservations

- **Global agents `master-code-architect`, `code-refactoring-specialist`, `solution-architect` (extend, not subsume):** preservation reason was "they fire only on explicit invocation and contain workflow detail beyond the single-line principle (multi-step architectural reasoning, refactoring-specific rules, structured clarification protocols)." That reason is weak because Explore #5 explicitly says it "did not enumerate the agent's contents beyond a one-line gloss" — the "deeper workflow detail" being preserved is asserted, not inspected. The scope-mapper is preserving primitives whose contents have not been read, on the assumption that depth-beyond-the-tagline exists. Worse, the preservation rationale ("fires only on explicit invocation") is an *argument for removal under the regret frame*: an agent that requires explicit invocation is an agent the user has to remember exists, which is the highest-friction form of guidance and the most likely to be forgotten when it matters. The preservation biases the downstream answer toward **agent-sprawl-as-default** — keeping seven global agents whose firing conditions and contents are unverified, while layering always-on instructions above them, accumulating exactly the "silent overlap-induced drift" the outside-view distillation flagged as the primary failure mode of the action being evaluated. If the always-on karpathy lines are added and the agents are kept, the user now has two layers covering similar ground with no measurement of which one fires when.

- **Default Claude Code system prompt — "no comments / no error handling for impossible cases" coverage (extend, not subsume):** preservation reason was "Anthropic owns it and the user cannot delete it; 'extend' is the only honest label." That reason is weak because *non-deletability is not a reason to extend* — it is a reason to leave alone and not duplicate. Treating "I can't delete it" as license to layer on top of it converts every Anthropic-owned default into an extension surface, which is precisely the accretion pathology the classifier's frame-bias note warned about. The canon distillation's contradiction #1 (minimalism vs. explicit-steering) and gap-flag (defaults are a moving target; "what defaults cover" should not be reasoned from memory) compound this: the scope-mapper is extending a primitive whose current contents are unknown and which the canon says might already do the job. The preservation biases the downstream answer toward **adding-because-we-can-not-delete**, which is structurally indistinguishable from the maximalist "import the whole plugin" position the prior recommendation was supposed to reject.

## Challenge to the classifier label

The classifier's alternative label was `investigation` (audit my reasoning trace) with the trigger "if no setup change is on the table regardless of outcome." The current frame quietly assumes a setup change *is* on the table (the action set includes "drop a line," "install globally," etc.), which keeps `extend` primary. But the meta-validation framing the user actually used — "should the prior recommendation stand, get rewritten, or get replanned?" — points at the *workflow's verdict*, not at a CLAUDE.md edit. Under the `investigation` label, the downstream frame would change substantially: the artifact under audit becomes the orchestrator's reasoning trace, not the karpathy plugin; the deliverable is a confidence statement about the prior recommendation (and the procedure that produced it), not a ranked action set; and "do nothing" stops being one option among many and becomes the *expected* outcome unless the audit surfaces a procedural flaw. The current `extend` framing imports an action-selection bias the user did not necessarily intend.

## Frame-level objection the orchestrator should carry forward

The regret-minimization frame mechanically favors the cheapest action, which is the prior recommendation; the generator must name what evidence — if it had surfaced in steps 3-7 — would have forced a recommendation *more expensive* than the prior one, and confirm whether that evidence was actually looked for or merely assumed absent.
