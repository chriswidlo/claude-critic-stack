# challenges.md — item 13 (orphans fate)

## Frame as given

The orchestrator reframed the user's "wire-in vs archive vs delete each of four orphans" into a two-step pipeline: **first categorize each orphan** (reusable artifact / frozen evidence / failed primitive), **then check whether an existing primitive already absorbs that category**, and only design something new for the residual gap. The implicit optimization shifted from "tidy repo" to "cheap reversibility of past judgment," with git history nominated as the venue for the frozen-evidence category. The scope-mapper then operationalized this by marking O1, O2, O3 as `replace` (delete; rely on git), O4 as `conflict` (defer to item 5), and the standing policy P as `subsume` (no new primitive — existing LEDGER states + session-artifact immutability + git history + the deliberately-deleted `temporary/` cover the space).

## Alternative frame

Reframe from *"categorize-then-route"* to *"the orphans are the policy."* Under this frame, the four documents are not specimens to be sorted into pre-existing bins; they are the *only* worked examples this repo will ever have of "what an expired-utility doc looks like in this codebase." Deleting them and writing a one-line rule (or no rule) optimizes for a *rule-shaped* artifact, but the repo's actual idiom — visible in CLAUDE.md, in the agents, in the punch-list itself — is *worked-example-shaped*: it shows rather than tells. Under this frame the verb changes from *dispose* to *exhibit*. The question becomes: "which of these four, kept in place with a one-line in-file header explaining their post-utility status, would teach a future maintainer (human or LLM session) more than their absence plus a git SHA would?" The optimization target flips from minimizing visible cruft to maximizing teaching surface per kilobyte. This is a different frame, not a different solution: it changes what counts as a good answer (a curated exhibit is good) and what counts as a bad answer (a clean repo with no in-tree memory of what was tried is bad).

## Condition under which the current frame is wrong

The current frame is wrong if a future LLM session, asked "why doesn't this repo have an archive folder / why was five-pressures replaced / why did warm-balloon ship the way it did," cannot reconstruct the answer in under two minutes from `git log` alone — because at that point git-as-venue has failed the job the frame assigned it, and the in-tree-exhibit frame would have been correct.

## Challenges to scope-map preservations

- **Git history as the venue for frozen evidence.** The stated reason is the user's own reframe sentence ("the right primitive is not an archive folder but git itself") plus the Anthropic 2025 hybrid-retrieval passage about maintained-index staleness. That reason is weak because it conflates two different jobs git does well: (a) recovering a known file by SHA, and (b) being *discoverable* to a reader who does not know the file ever existed. Git is excellent at (a) and structurally bad at (b) — `git log -- prompts/five-pressures.md` requires you to already know the path. The scope-mapper imported the Anthropic passage about glob/grep beating maintained indexes, but glob/grep work on the *current* tree; once a file is deleted, it is invisible to both. The preservation biases the downstream answer toward *deletion-with-confidence* and away from any in-tree breadcrumb (a one-line note in CLAUDE.md naming the deleted artifacts, a `RETIRED.md` in `prompts/`, or a comment block in the punch-list itself), even though such a breadcrumb costs ~3 lines and resolves the discoverability gap git cannot.

- **The deleted `temporary/` folder (preserved-as-deleted).** The stated reason is that recreating it under any synonym (`archive/`, `attic/`, `retired/`) would re-introduce what item 6 just removed. That reason is weak because item 6 deleted `temporary/` for a *specific* failure mode (it became a junk drawer with no exit criteria), and the scope-mapper is generalizing that specific failure into a blanket prohibition on any "stuff that isn't quite live" venue. The two failure modes are not the same: a *tagged* in-place state (e.g., a header line `Status: retired 2026-04-27, see git log <sha>`) has exit criteria built in (the file is its own record), whereas `temporary/` had none. The preservation biases the downstream answer toward the false dichotomy "either fully live or fully deleted," which suppresses the in-place-with-status-header option that several of the orphans (especially O3, the warm-balloon plan) are best suited for.

- **One-line rule in CLAUDE.md / decision-log as the only permitted policy-shaped artifact.** The stated reason is the outside-view's mitigation ("Default for orphans is delete; archive only for historical value; wire-in only if load-bearing") at ~30 seconds of work. That reason is weak because the outside-view itself flagged in its own gaps that "the cost of the one-line rule itself becoming stale or contradicted by future decisions" was *not* examined — the mitigation was treated as free. A one-line rule in CLAUDE.md is a load-bearing constraint on every future session and inherits all the failure modes of any unmaintained policy text; the scope-mapper imported the outside-view's mitigation without inheriting the outside-view's own caveat. The preservation biases the downstream answer toward *writing a rule* rather than *demonstrating a default by example* (e.g., a single committed message on the orphan-deletion PR, with no rule promoted to CLAUDE.md at all).

## Per-orphan pushbacks the scope-mapper under-argued

- **O1 (five-pressures.md) — is `replace` over-reach?** The scope-mapper claims the 12-step workflow "absorbed" the 37-line LLM-failure checklist via frame-challenger + critic-panel + outside-view. Test that claim: five-pressures was a *static, scannable, in-one-glance* checklist; the 12-step workflow is a *dynamic, multi-agent, artifact-producing* pipeline. These are not the same primitive at different sizes — they are different shapes. A pre-flight checklist and a multi-stage review pipeline coexist in aviation, in surgery, and in code review (linters + PR review). The scope-mapper's "absorbed" framing presumes shape-equivalence and does not defend it. If even one of the five pressures is not redundantly enforced by a critic lens (the scope-mapper's own deletion-cost note hedges this: "if any specific failure mode is not redundantly covered … that gap is the one thing worth porting"), then `replace` is over-reach and the correct verb is `extract-then-replace` — and the extraction work is not done.

- **O3 (warm-balloon plan) — is `replace` over-reach?** The scope-mapper says the plan is "fully shipped per git log" and the rationale "lives in commit messages." Test that: commit messages are written for the immediate reviewer, not for a six-months-later "why did we choose this Phase-2 shape" reader. The warm-balloon plan is the only document in the repo that captures the *alternatives that were not taken* in Phase-2 — that information is structurally absent from the shipped code and from the commit messages (which describe what *was* done, not what *was rejected*). Preservation as a precedent / lineage doc (kept in `plans/` with a header `Status: shipped 2026-04-27, retained as lineage`) is not the same primitive as `plans/` for in-flight items, and the scope-mapper dismissed this option without naming it. The `replace` verdict treats "shipped" as "discardable," which is the warehouse-inventory frame, not the design-record frame.

- **P (the standing policy) — is `subsume` papering over a gap?** The scope-mapper enumerates four existing primitives (LEDGER states, session-artifact immutability, git history, the absent `temporary/`) and concludes they cover the lifecycle space. Test that by application: which of those four primitives covers a 37-line prompt checklist whose function has been partially absorbed by a pipeline? LEDGER is for upgrades (prospective work, not retrospective artifacts); session-artifacts are immutable but only for *session* records; git history covers recovery but not discoverability; `temporary/`'s absence covers nothing — it is the *lack* of a venue. The scope-mapper's "combination of four" is a list of primitives that each cover a *different* slice; what the orphans expose is the slice none of them cover ("a load-bearing instruction artifact whose load-bearingness has decayed"). The default-disfavors-new-primitives rule is doing real work here, possibly too much: the honest read is "no existing primitive covers this, but the recurring cost is below threshold," not "existing primitives cover this." Those are different claims and the scope-mapper conflated them.

## Challenge to the classifier label

The classifier landed on `new` (policy is a new primitive) and rejected `refactor` because the punch-list "explicitly invites a standing policy." That rejection is too quick. If the generator concludes (as the scope-map already hints) that no new primitive is warranted and the right move is to delete three files and defer the fourth, the operation that was actually performed is *refactor* (move/delete files without changing what the repo can do). Under the `refactor` framing, the question is not "what policy should we write" but "what is the smallest tree-delta that removes the orphans without losing information"; the policy layer evaporates entirely, and the per-orphan calls become the whole question. The downstream frame would change in one specific way: the generator would not be obligated to produce a policy-shaped recommendation at all, and the "ways this could be wrong" section would shift from "the policy might be over-engineered" to "the deletion might lose information that wasn't in the commit message."

## Frame-level objection the orchestrator should carry forward

The whole "categorize-then-route-to-existing-venue" pipeline assumes git history is a venue for *discoverability* and not just *recovery* — if it isn't, every `replace` verdict in the scope-map silently degrades the repo's ability to answer "why did we do it this way" six months from now, and no policy-layer decision can repair that.
