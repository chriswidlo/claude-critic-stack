# Frame challenges — format-only state-transition gate

## Frame as given

The orchestrator's Revision 1 frame treats this as **enforcement-mechanism design over a fixed lifecycle**: the 8-state spine is the contract, and the open question is what shape the enforcer takes (script vs agent, hook vs slash command, schema-first vs gate-first, targeted vs uniform). The implicit optimization target is *format consistency at scale* — the lab grows from 22 entries to 100+ and the state column must stay meaningful under heterogeneous AI authorship. Both alternatives the orchestrator surfaced (schema-first; targeted on `🔨` and `💎`) still presume the artifact-to-build is a *validator*. The artifact-to-build is the load-bearing assumption that goes unchallenged.

## Alternative frame

**The 8-state lifecycle is a measurement instrument that has not yet been measured.** Reframe: the question is not "how do we enforce the lifecycle" but "is the lifecycle accurate enough to enforce?" Under this frame, the verb is **calibrate**, not **gate**. The first deliverable is an audit report over the existing 22 entries — for each entry, what state did the operator believe it was in, what evidence is in the body, and where do those two diverge? That report is the cheapest experiment that tells you whether you have a drift problem worth gating, a *spec* problem (the 8 states don't carve reality at the joints), or a *workflow* problem (the operator never advances anything past `🌱`, which most personal-tooling corpora exhibit). Only after the audit does the design space include a gate at all; the audit may instead recommend collapsing the lifecycle, deleting unused states, or moving to a 1-bit "open/closed" status with everything else in prose. The user pre-decided that the answer is a gate; this frame refuses that pre-decision and asks what the corpus is actually telling you.

## Condition under which the current frame is wrong

The current frame is wrong if, when you actually inspect the 22 existing entries, fewer than three of them have ever advanced past `🌱 created` — because then the modal failure mode is not drift but disuse, and a gate on transitions that almost never happen is enforcement infrastructure for a non-event.

## Challenges to scope-map preservations

- **The 8-state lifecycle (extend, stated reason: "the user's framing presumes the existing lifecycle is the contract being enforced"):** preservation reason was *the user's framing presumes it*. That reason is weak because the outside-view distillation explicitly says "8 states is on the high end for personal lifecycle; most surviving personal workflows collapse to 3–5 states within a year" and names state-collapse as failure mode #2 ("validator enforces lifecycle operator no longer believes in"). The strongest argument for collapsing the lifecycle to 3–5 states *before* building any gate is: **you are about to write a per-state schema for seven transitions, half of which the corpus may never reach, and the schema-writing cost is paid once but the schema-maintenance-as-states-collapse cost is paid forever in spec drift** (which the same distillation names as failure mode #1, the *modal* failure). Collapsing first means the gate's surface area is 2–4 transitions instead of seven; it means the README's "lifecycle is descriptive, not prescriptive" sentence (a live conflict in the scope-map) becomes easier to honor because there are fewer states to be prescriptive about; and it means the load-bearing transitions (`🔨 implemented`, `💎 value-proved`) are not buried inside ceremonial ones (`✅ accepted`, `⚙️ run-through-repo`) that may exist only because the lab was designed before any entry had been advanced. Preserving 8 states biases the downstream answer toward **a per-state schema that will lag the operator's actual writing template within months**, because the spec was set before the writing pattern stabilized. The honest move is: ship nothing until you have collapsed the lifecycle to whatever the existing 22 entries demonstrate it actually needs.

- **The state column in entries (extend, stated reason: "the user has not chosen the folder-as-state alternative"):** preservation reason was *user hasn't picked the alternative*. That reason is weak because the canon-librarian distillation surfaced Anthropic 2025's "folder hierarchies, naming conventions, and timestamps all provide important signals" passage as a *direct* contradicting passage, and the implication is sharper than the scope-map credits: **if file location is the state, the gate problem dissolves into a `mv` operation whose precondition is a presence-of-section check** — no state-table parsing, no tolerant matching for `## Spike` vs `### Spike Notes`, no schema config file, no per-state validator at all. The preservation biases the downstream answer toward **building a parser for a structure (the in-entry state table) whose existence is the very thing being questioned by the corpus the librarian retrieved**. The README's anti-pattern §"Grouping by status" defends in-entry state on philosophical grounds, but that anti-pattern was written before the gate existed; if the gate is new infrastructure, the anti-pattern is up for revision too. Preserving the state column quietly closes off the cheapest possible gate shape.

- **`/upgrade` slash command capture path (extend, stated reason: "operator-typed `/upgrade <thought>` would break"):** the reason is reasonable on its face but biases the downstream answer toward **a sibling `/advance-upgrade` command rather than a subcommand inside `/upgrade`**, which the scope-map itself flags as an unresolved conflict. Preservation of the capture path was never in question — what was in question is whether advancement and capture share a command surface. Treating the capture path as preserved-untouched obscures that the *real* preservation decision is whether `/upgrade` stays a single-purpose command or becomes a multi-verb command. That is a bigger UX commitment than the scope-map names.

- **Per-tier folder structure (extend, stated reason: "tier and state are orthogonal"):** the reason is correct *if* state stays in the entry. If the alternative frame above (audit first) recommends collapsing state into folder location, tier and state stop being orthogonal because both want to live in the path. Preserving per-tier folders biases against any folder-as-state design without ever forcing that tradeoff into the open.

## Steel-man for uniform gating (counter to frame.md's targeted-gating alternative)

Frame.md Revision 1 alternative #2 argued the failure mode is concentrated on `🔨 implemented` and `💎 value-proved` and the other five transitions are either trivial or operator-self-gated. Steel-manning the uniform position: **uniform gating is the right shape when the cost of two-tier rules exceeds the cost of writing trivial schemas for the trivial transitions.** The two-tier cost is paid in three currencies the targeted frame undercounts. (1) **Cognitive overhead at the call site** — every operator and every AI agent now has to remember which transitions gate and which don't, which is exactly the "operator memory" failure mode the outside-view distillation names as bypass-by-default; uniform rules require zero memory ("everything goes through the same door"). (2) **Schema-evolution asymmetry** — the day a "trivial" transition stops being trivial (e.g., `📋 → ✅` grows a "must reference the workflow synthesis" requirement because operator self-gating proved unreliable), the targeted system needs a control-flow change *and* a schema addition, while the uniform system needs only a schema addition; uniform rules push all change into the data, which is the cheaper place to change things. (3) **Precedent for the next gate** — if a second gate is ever needed (lifecycle for some other artifact, e.g., `limitations.md` entries), the uniform pattern is copy-pasteable; the targeted pattern requires rejudging which transitions matter every time, which is judgment work that AI agents cannot reliably do. Uniform gating would be the right shape *if and only if* (a) the per-transition schemas for the "trivial" five are genuinely cheap (a `## Spike` heading check is one line), (b) the operator commits to lockstep schema-revision discipline so trivial schemas don't ossify, and (c) the gate has a single denial UX that doesn't bifurcate into "soft warning for the easy ones, hard block for the hard ones." If those hold, the cost of uniform rules is roughly seven one-line presence checks; the cost of two-tier rules is permanent cognitive overhead and a worse story for the second gate. The targeted frame is right that the *risk* is concentrated; it is wrong to conclude that the *rules* should be. Worth carrying forward as live tension; the generator should not silently default to targeted because the frame named it second.

## Challenge to the classifier label

The classifier's primary label was `new` with `extend` as alternative; the alternative becomes substantively different if the operator decides the gate is *enforcement of an already-defined lifecycle*, in which case the artifact is not a new system but a mechanical reading of the README. The downstream frame under `extend` would be considerably smaller: no per-transition schema design (the README is the schema), no schema-source-of-truth file to debate, no "tolerant matching" implementation work, and no decision about whether the schema lives in YAML or in the script. The gate becomes a literal regex pass over the README's prose for "required" language, which collapses the design space from a 2×2 of pre-decided axes to a single question: *"can a script read the existing README and produce yes/no?"* The orchestrator's frame inherits the `new` bias and is sized for a system; under `extend`, the same problem is sized for an afternoon. The classifier flagged this in its own gaps section ("is the schema authoritative or descriptive?") and the answer to that question is the actual classification fork.

## Frame-level objection the orchestrator should carry forward

Build the audit before the gate — without evidence that the existing 22 entries exhibit drift the gate would have caught, the gate is enforcement infrastructure for a hypothetical failure mode and will hit the SRE 2016 abandonment threshold before it has prevented anything.
