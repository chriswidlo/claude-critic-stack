# Frame challenges — review-diagnostics-impl

## Frame as given

The orchestrator's Revision 1 reframes the user's *did-we-build-it-right* question as **instrument-fit + dogfooding-as-confounder**: optimize the review for whether the shipped layer actually answers the operator's four catalyst questions (especially "where did the 10 minutes go?"), and treat this very session's metrics as a sample-of-one drawn from the easiest possible workload, so panel findings are not over-anchored to it. The frame keeps AI-blindness as a load-bearing invariant, leaves the shipped state as a floor only by panel consent, and counter-pressures research-theater bias by demanding each finding map to a concrete action.

## Alternative frame

**Workflow economics, not workflow observability.** The instrument-fit reframe asks "does the layer measure what the operator asked to measure?" That is a measurement-quality question. The harder question — and the one that determines whether the layer is decoration or load-bearing — is *what decision does any of this data change?* Reframe the review around: for each metric the layer emits (`duration_seconds`, `tokens_by_agent`, `verdicts`, `loop_count`, the four-field ledger), name the workflow step that would be altered if the metric crossed a named threshold. If `tokens_by_agent` shows critic-architecture dominating at 3× the next lens, what step changes — do we re-prompt, re-weight, retire the lens? If `loop_count` averages 1.8 across N sessions, what does the orchestrator do differently at step 11? If no answer survives, the metric is decoration regardless of how accurately it is measured, and the canon §As Simple as Possible passage applies: candidates for removal, not for tightening. Under this frame the review's verb changes from *validate-and-extend* to *decide-which-metrics-survive*; the panel's job is to prune, not to queue SOTA.

## Condition under which the current frame is wrong

The current frame is wrong if the operator-question list ("where did the 10 minutes go," "what cost most tokens," "are critics balanced," "did one agent dominate") is itself research-theater self-justification rather than a real diagnostic need — i.e., if the operator has never, in practice, changed a workflow step in response to any answer to those questions and would not change one even if the layer answered all four perfectly. In that case "instrument-fit" is the wrong optimization target because the instrument has no consumer; the frame is solving for a category-error and the panel will dutifully tighten regexes and queue per-step timing for a dashboard that nobody opens (outside-view's tertiary failure mode at ~3 months, exactly).

A second, narrower falsifier: if session count over the diagnostics layer's first six months is plausibly under ~20 — explore §10 confirms zero populated `diagnostics/` folders today, and the canon librarian distillation cites Anthropic's own ≥20-queries-for-evaluation guidance — then averages, percentile shapes, and cross-session aggregations are pre-empirical and the SOTA wishlist is queued against a dataset that may never reach statistical relevance. The frame's implicit "more sessions will accumulate, then we'll have signal" is an assumption the panel has not stress-tested.

## Challenges to scope-map preservations

- **Row 6 — schema doc `<target>/.claude/session-artifacts/README.md`: preservation reason was "doc was updated in the same change-set as the implementation and explore §6 confirms additive sections are present and consistent; no drift surfaced."** That reason is weak because *no drift can have surfaced yet* — explore §10 confirms zero completed sessions have populated `diagnostics/` folders, so there is no consumer-side execution path through the documented schema against which drift could appear. "No drift detected" against a never-exercised consumer is not evidence of fitness; it is evidence that the test conditions have not yet existed. More specifically: the schema doc now treats the `## Metrics` block (the diagnostics-emitted suffix) as **additive and optional**, but the doc does not declare which fields inside that block are load-bearing for ledger-render's render contract versus which are decorative. If `ledger-render` silently degrades when `metrics.json` is missing (explore §3, line 71) but renders confidently when `metrics.json` is *present-but-malformed* — say, parser produced a `metrics.json` with a stale schema after a host-product layout change — the schema doc gives no producer/consumer conformance contract to catch the gap. The preservation biases the downstream answer toward **"doc is fine, problem is in the parser/wrapper"**, which lets the review queue regex tightening and `hook-exec.sh` work while leaving a load-bearing-by-omission claim ("`## Metrics` is purely additive, no schema obligations on its contents") untested. That is a deferred problem dressed as a free lunch: the first time a consumer (ledger-render or explain) breaks because a `metrics.json` field disappeared or changed type, the bug will be attributed to the parser, but the root cause will be that the schema doc never specified what the consumers were entitled to assume. Subsume here is the comfortable call; the honest call is `extend` — the doc must grow a producer/consumer schema-conformance section before subsumption is defensible.

## Challenge to the classifier label

The classifier returned `investigation` primary with `extend` secondary and the explicit hedge that `extend` becomes primary "if the user's actual ask is 'decide which SOTA upgrade ships next' rather than 'validate what shipped.'" The user's framing words include "identify SOTA upgrades worth queueing" — present-tense queueing, not past-tense validation — and `scope-map.md` already produces a *ranked three-action list* (resolve row 10 conflict; replace `2>/dev/null` with `hook-exec.sh`; tighten regexes + add schema-conformance test). A ranked three-action list is the output shape of an `extend` session, not an `investigation` session; investigations enumerate, extensions prioritize and commit. Under `extend` as primary, the frame would shift from "validate-the-shipped-floor + queue-the-backlog" to "the diagnostics layer is the primitive being grown, and the review's job is to commit to which growth lands first and which is rejected as not-yet-justified." Two consequences: (a) the frame can no longer hold the shipped state as a floor by default — partial revert (e.g., disable the parser until row 10 is resolved) becomes a first-class candidate alongside extension, because under `extend` the question "does this primitive's contract grow or shrink?" is symmetric, and (b) the panel would be obligated to reject SOTA candidates with named reasons rather than list them, which is the canon §As Simple as Possible discipline applied at the frame level. The current `investigation` framing lets the panel queue without rejecting; the alternative `extend` framing forces the panel to either ship a SOTA item or kill it. Given that explore §10 shows the layer has zero real-session evidence to draw on, an `extend` frame would likely conclude *kill more than it ships*, which is the opposite default from what the current `investigation` frame produces.

## Frame-level objection the orchestrator should carry forward

For each metric the diagnostics layer emits, name the workflow decision it changes; metrics with no named decision are candidates for removal under canon §As Simple as Possible, not candidates for SOTA tightening — and the candidate must explicitly list which (if any) shipped fields fail this test rather than treating all of them as load-bearing by default.
