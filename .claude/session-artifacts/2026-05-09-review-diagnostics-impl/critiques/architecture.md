# Architecture critique — review-diagnostics-impl

## 1. Weakest structural link

**The parser is a single module that crosses three aggregate boundaries and owns four distinct responsibilities.** `<target>/bin/parse-session-metrics.py` (257 lines) walks (a) the host-runtime transcript layout under `~/.claude/projects/<encoded-cwd>/<uuid>.jsonl`, (b) the host-runtime sub-agent layout under `<transcript>/subagents/agent-*.meta.json` with sibling `.jsonl`, (c) the workflow's `critiques/*.md` markdown, and (d) the workflow's `decision-log.md` prose. Four different schemas, three different aggregate owners (Anthropic Claude Code product, the parser-itself, the orchestrator workflow), one module. The candidate's actions 1–6 harden three of those surfaces but never name that the module itself is the wrong shape: it should be at minimum two modules with one stable contract between them — *transcript-attribution* (host-coupled, canary-guarded) and *workflow-artifact-summary* (workflow-coupled, schema-stable). Killing the `verdicts` parser (action 1) and tightening loops (action 7-deferred) only lower the count of responsibilities from four to three; the boundary violation persists.

This matters because the failure modes are non-uniform across the boundaries: host-layout drift is silent and external; verdict-regex looseness is silent and internal; workflow-prose drift is observable. Bundling them into one process means the canary in action 6 has to disambiguate which boundary failed — and the proposed canary (`transcript-found: yes/no`, `subagent-count: N`, `total-tokens: N`) only covers boundary (a). A canary scoped to the union of four responsibilities cannot tell the operator which boundary actually broke.

## 2. Invariants at risk (not named in the candidate)

1. **Token-attribution monotonicity under partial failure.** If `find_subagents_dir` returns `None` while `find_transcript` returns a path (line 33–36 of the parser), `tokens.total` is the sum of *only the main thread*, but `tokens.by_agent.main` is reported as a complete row. A consumer comparing `total` against `sum(by_agent.values())` gets equality and concludes data is intact — when in fact every sub-agent was dropped silently. The candidate's canary (`subagent-count: N`) catches `N == 0`, but does not catch the case where the subagents dir partially exists (e.g., 2 of 5 `agent-*.meta.json` files have malformed JSON; line 110 silently `continue`s).
2. **Critique file naming as schema.** `parse_verdicts` (line 175–184) excludes any stem containing `"shadow"` or `"comparison"`. This makes the *file naming convention* a parser-load-bearing schema — if a future lens is added named `critic-shadowfax`, it will be silently excluded. The convention is undocumented anywhere in the candidate's proposed conformance section. Killing the `verdicts` field (action 1) eliminates this — but only if no future field re-introduces a name-based filter.
3. **Background-process atomicity.** `stop.sh:27` backgrounds the parser with `nohup ... &`. There is no fsync or rename-into-place guard around `metrics.json.write_text` at line 251. A second `Stop` event for the same session (e.g., user resumes and re-stops) races against the first parser instance writing the same file. The candidate's canary writes `diagnostics/canary.txt` on every Stop — same race, same module, no atomic-write discipline.
4. **`.meta.json` parsing as trust boundary.** Line 109 decodes a JSON file produced by Claude Code internals; `agentType` (line 112) is taken at face value as a key into the orchestrator-facing `tokens.by_agent` map. If Claude Code ever introduces an `agentType` containing a `/`, `.`, or whitespace, the value flows unsanitized into `metrics.json` and from there into ledger-render's YAML dump (line 72 of ledger-render SKILL). Schema conformance contract (action 4) does not address this — it specifies what consumers may *assume* about fields, not what producers must *sanitize* at the host-runtime trust boundary.
5. **The `<3 sessions` fallback is a hidden invariant.** Explain skill (line 41) silently substitutes `~10 min` when fewer than three populated `metrics.json` files exist. The candidate addresses the `duration_seconds`-as-rendered concern by demoting it from the explain card, but the threshold itself (3) is hardcoded at `<target>/.claude/skills/explain/SKILL.md:41` and not in the conformance section. Any consumer-side fallback threshold belongs in the conformance contract or it is invisible policy.

## 3. Coupling and direction

- **Volatile → stable, intended:** workflow-artifact consumers (ledger-render, explain) → `metrics.json` schema. The proposed conformance section (action 4) makes this explicit, which is correct.
- **Stable → volatile, problematic:** the workflow (stable) depends, transitively through the parser, on the host-runtime transcript layout (volatile, owned by another team, undocumented). This is the inversion canon §loose-coupling names. The candidate's response (canary + accept-coupling) treats this as a posture choice; architecturally it is a layering violation that should be confined to a single adapter module rather than diffused through the parser's main function.
- **Cycle:** none in the runtime call graph. But there is a *documentation cycle* the conformance section will introduce: `metrics.json` schema doc cites the parser as its producer; the parser's correctness depends on host transcript layout (undocumented elsewhere); the canary's contract depends on a host-runtime fact the schema doc cannot pin down. The conformance contract pretends to be a closed system but rests on an open one.
- **Layering violation:** `parse_verdicts` reads `<workflow>/critiques/*.md` from a process spawned out of a host-runtime hook, not out of the workflow. The hook trio is a *host-runtime* concern (it runs on every Claude Code Stop event, regardless of whether a 12-step workflow is in progress); the parser reaches *up* into workflow-artifact territory to extract verdicts. Killing the `verdicts` field (action 1) actually *fixes* this layering violation, more than the candidate credits — that's an under-stated benefit of action 1 that should be made explicit.

## 4. Ignored architectural alternatives

1. **Split the parser into a host-adapter and a workflow-summarizer, communicating via a stable intermediate JSON.** The host-adapter writes `diagnostics/raw-attribution.json` (token totals, agent map, tool counts) — its boundary is the host transcript. The workflow-summarizer reads that JSON plus workflow artifacts and writes `metrics.json`. The conformance contract (action 4) lives on `metrics.json` cleanly because the workflow-summarizer is the only producer; the canary lives on the host-adapter; failures attribute to one boundary. This is the right shape under canon §loose-coupling and the candidate does not consider it.
2. **Make the parser a pull, not a push.** Today the parser is fired by `stop.sh` as a side effect, fire-and-forget, with a race window against ledger-render (explore §17). Inverting this — `metrics.json` is computed lazily by ledger-render itself when invoked at step 13, by calling a parsing library, not a script — eliminates the race, eliminates the staging→diagnostics directory dance, and collapses the canary to "did the library raise." The candidate's action 6 (canary) is a workaround for a push-architecture defect that a pull-architecture would not have. The cost is that quick-take and bypass sessions don't compute metrics — which is the correct behavior anyway (per the bypass policy).
3. **Use the host-runtime's structured event hooks, not the JSONL transcript.** Claude Code's `PostToolUse` hook receives structured event payloads. The parser could accumulate token usage from those events directly into `events.jsonl` during the session (rather than re-reading the transcript at Stop), eliminating the dependency on the undocumented `<encoded-cwd>/<uuid>.jsonl` layout altogether. This requires verifying that hook payloads carry the usage data; if they do, the modal failure mode (host layout drift) collapses to "host hook payload schema drift" — a smaller, more contractual surface. The candidate explicitly considered and rejected option (b) "defer until stable API" but did not consider the hook-event payload as a candidate stable surface.

## 5. Frame-level objection

**The candidate frames the architecture as "what to keep, what to kill, what to defer" — but the diagnostics layer's primary architectural defect is that it is a *single module spanning three aggregates*, and none of the candidate's seven actions address that boundary.** Actions 1, 4, and 7 reduce the parser's output surface. Actions 2 and 6 add structural enforcement around a sub-aggregate (hook AI-blindness; transcript-found canary). Actions 3 and 5 are non-architectural. The diagnostics layer's *aggregate boundaries* — and which module owns each — is the unanswered question. A reader of the candidate would conclude that the parser is fine in shape and just over-emits; an architecturally honest reading is that the parser is wrong in shape, and the candidate's pruning is local-consistency improvement on a structure that should be split.

This is frame-level, not lens-specific, because the candidate's *organizing question* ("what survives the workflow-decision-named test?") is field-by-field, not module-by-module. A module-level reframe ("for each aggregate this layer touches, name the owner, the contract, and the failure mode") would have surfaced the split-the-parser action as the highest-leverage architectural change, and demoted action 6 (canary) to a stopgap on the new sub-module.

## 6. Lens findings — direct answers to the stress questions

**Q1 — `hook-exec.sh` wrapper, single point of failure.** The candidate's "shift silent → noisy" trade is *architecturally sound but under-specified.* A wrapper that fails noisily means: when the wrapper itself has a bug, *every* hook fails on every Claude Code event, which is immediately observable to the user (Claude Code surfaces hook errors). When the convention has a bug (a future `echo "DEBUG"` without redirect), *one* AI context is corrupted silently, which may never be observed. Trading silent-narrow for noisy-broad is the right call when the noisy failure is bounded — but the candidate does not specify the wrapper's *recovery* shape. Concretely: if `hook-exec.sh` itself is missing or non-executable at session start, does `session-start.sh` fail-open (proceed without diagnostics) or fail-closed (block the session)? The candidate does not say. Without that, "noisy" could become "session-blocking," which is worse than the convention. **Action item:** specify wrapper failure semantics — fail-open with a one-line stderr to the host hook log, never to AI context, and a fall-back to the convention if the wrapper is absent.

**Q2 — killing the `verdicts` field, consumer migration cost.** Verified: ledger-render at SKILL.md:70–72 reads `metrics.json` and dumps it whole as YAML — it does *not* read the `verdicts` field semantically, it just forwards it into the rendered ledger. Explain SKILL.md:33–46 only reads `duration_seconds`. **No consumer parses the `verdicts` field.** Migration cost is exactly: delete lines 170–185 and line 248 of the parser, delete one schema entry from the README. The candidate's "~30 min" estimate is correct. The risk the candidate names ("future cross-session verdict-pattern dashboard" closes a door) is real but trivially reversible, and the right re-entry is *structured frontmatter in critiques/*.md*, not a regex — which the candidate already names. **Approve action 1 unreservedly.**

**Q3 — schema doc as conformance contract vs. JSON Schema file.** Markdown prose is the wrong shape for a conformance contract. The candidate proposes a "Metrics conformance" prose section in the schema doc; that is better than nothing but does not gain the property a real conformance contract should have: *executable validation*. A `metrics.schema.json` file (JSON Schema draft-2020-12, ~30 lines) plus a one-line `jsonschema --instance metrics.json --schema metrics.schema.json` check in the parser tail and in ledger-render's pre-read would catch every drift case the conformance section is meant to catch, mechanically. Prose conformance sections decay; schema files do not. **Action item:** replace action 4 with "add `metrics.schema.json`; parser validates output before write; ledger-render and explain validate input before read; on validation failure, both consumers silent-skip per the existing missing-file policy." Prose section in the README cites the schema file as authoritative.

**Q4 — canary-with-acceptance vs. coupling debt.** Architecturally, "canary-with-acceptance" is *postponing coupling debt*, not paying it. Canon §loose-coupling is unambiguous: integrate against stable APIs. The right architectural posture is alternative #3 above (use the host's structured hook events, which are *contractual*) or alternative #1 (split the parser so the coupling lives in one named adapter module, isolatable when the host layout shifts). Canary-with-acceptance is fine as a *transitional* posture, but only if a deletion path is named. The candidate's action 5 (Heilmeier exam at 6 months) is a *value-prove* exam — not a *coupling-debt-paydown* exam. These are different. The diagnostics layer needs *both*: "did the layer earn its keep" and "is the parser still talking to the same layout it shipped against, or has it accumulated three workarounds." Add a second exam: at 6 months, count parser commits that target host-layout-drift. If non-zero, the layer's coupling debt has been called and the layer enters a redesign review (replace, not patch).

**Q5 — aggregate boundaries.** Already answered in section 1: the parser is doing too much for one module. The host-transcript adapter, the workflow-artifact summarizer, and the verdict/loop scraper are three modules with three different change frequencies and three different failure modes. The candidate's pruning leaves the verdict scraper killed and the loop scraper deferred — which collapses the four responsibilities to two. That is acceptable as a transitional state but should be named as transitional, not as the end-state.

## 7. Verdict

**rework**

What would change the verdict to `approve`: replace action 4 with a `metrics.schema.json` JSON Schema file plus producer/consumer validation, specify wrapper failure semantics in action 2 (fail-open with named fallback to convention), and add a second Heilmeier-style exam in action 5 covering coupling-debt-paydown alongside value-proved. The split-the-parser alternative is architecturally preferred but accepting the candidate's monolith with these three changes is a defensible transitional architecture.

Leading indicator that this rework is going wrong: if the conformance contract lands as prose and within three months a `metrics.json` field gets renamed in the parser without a corresponding consumer-side error, the contract is decoration and the redesign-into-two-modules call should be made.
