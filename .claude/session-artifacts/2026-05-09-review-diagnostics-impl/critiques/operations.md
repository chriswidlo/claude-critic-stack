# Critic — Operations lens — `2026-05-09-review-diagnostics-impl`

## 1. Most likely incident

At time T (a Stop hook firing on a long workflow session that produced ledger-render output), the parser's `nohup ... &` background invocation had not yet written `metrics.json` when ledger-render read for it; the ledger silently rendered without a `## Metrics` block, was committed, and the session entered the audit history with empty diagnostics. Root cause: the candidate proposes no synchronization between the Stop hook's fire-and-forget parser and any downstream metrics consumer, and silent-skip is treated as a safe default rather than as a failure-detection gap. The same incident at T+N days produces a staging directory with N orphan UUIDs and a canary that does not fire because the canary is also written by the parser that did not run.

## 2. Blast radius

- **Per-session radius:** one ledger entry. The session looks fine; the ledger is just thinner. Modal failure is invisible to the operator at the moment it occurs.
- **Cumulative radius:** every session run between failure-onset and detection. The candidate's canary triggers at three consecutive failures, so the *minimum guaranteed* blast radius is three sessions plus the canary's review latency.
- **Orphan staging radius:** unable to estimate steady-state size of `<target>/.claude/.metrics/staging/` and `<target>/.claude/.metrics/orphan/`. Explore §11 already shows 4 staging + 4 orphan UUIDs from pre-handoff sessions; the candidate adds no cleanup policy. Growth rate is one directory per Stop hook minus one per resolved `workflow-id.txt`. There is no named retention bound.
- **Cross-session radius:** zero — diagnostics are per-session; one corrupt `metrics.json` does not poison another session's parser run. That bound is real and worth keeping.
- **Cross-machine radius:** the parser's coupling to a user-home transcript layout means a single Anthropic-side layout change zeros every machine running this stack at once. One vendor commit, global silent failure. The candidate acknowledges this in its assumption #3 but does not propose a defensive measure beyond the action-6 canary.

## 3. Rollout / rollback

- **Rollout strategy:** the candidate names six actions but no rollout sequencing. Action 1 (kill verdicts field) and action 2 (`hook-exec.sh` wrapper) both touch live hook/parser code. There is no canary deploy, no shadow lane, no flag. The proposal is "land and watch the next session."
- **Rollback for action 2 (`hook-exec.sh` wrapper):** the candidate's stated rollback is implicit — delete the wrapper, restore inline `2>/dev/null` redirects to three hook scripts. This is **slow** under stress, not fast. The redirects were originally placed by hand (Explore §1: every external command in three scripts), and re-debugging which lines need redirection while a leak is corrupting an active critic run is exactly the wrong stress profile. There is no version-controlled snapshot of the pre-wrapper hooks named in the candidate. **Recommendation: introduce the wrapper additively first** — the three hooks still use inline redirects *and* call the wrapper for new commands — let two-systems-running run for at least 5 sessions, then cut over. The candidate currently proposes a hard cut.
- **Rollback for action 1 (kill verdicts):** lossy. Once the `verdicts` field is removed from the schema and ledger-render stops reading it, restoring it requires re-parsing historical critiques. The candidate accepts this under tradeoff #2 but does not name a deprecation window. Operationally, "delete on land" with no n-session deprecation window means any consumer not yet identified breaks instantly.
- **Two-systems-running window:** the candidate names none. Cost during overlap is unaddressed.

## 4. Observability gap

The candidate makes the *parser failure mode* invisible by design and proposes the canary (action 6) as compensation. Several gaps remain:

- **Parser-ran-but-produced-empty-`metrics.json`** is treated identically to **parser-did-not-run**. The Stop hook backgrounds the parser fire-and-forget (Explore §14, hook line 27); there is no exit-code capture, no "parser PID completed" sigil, no "parser write attempted at $(date)" log. The canary, written *by the parser*, cannot distinguish parser-crashed from parser-ran-and-found-nothing. The operator's debugging story for "why is `## Metrics` missing" is purely forensic.
- **Time-to-detect:** the canary fires after 3 consecutive empty sessions. At one session per week (the more conservative end of operator cadence), MTTD is ~21 days. At one per day, ~3 days. The candidate does not name the regime it is designing for.
- **Whether `metrics.json` exists at ledger-render time** is not measured. Ledger-render's silent-skip (Explore §3 line 71) is the right *fallback* but is not a *signal*. The proposal collects no data on how often the silent-skip path fires, which means the operator has no evidence base on whether the race is theoretical or routine.
- **Whether any metric is collected for the metric collector itself.** Parser duration, parser exit status, parser staging-dir-cleanup rate — none of these are named in the candidate. From Explore §2 and the Stop hook, my read is "no, none are."

The metric that would confirm the proposal succeeded — "at least one orchestrator step changed in response to a metrics reading within 6 months" (action 5's Heilmeier exam) — is itself not collected automatically. There is no review trigger, no calendar reminder, no count. Whether this is collected depends on whether someone reads the upgrade entry on month six. I would call that "uncollected."

## 5. Cost at failure

- **Modal failure (parser silently produces no `metrics.json`):** operator confusion on the next `/explain` invocation; ledger entry without metrics block; 3-session lag before canary fires; staging dir bloats by N orphaned UUIDs at one-per-session because the resolved-workflow-id branch's `rm -rf "$STAGING"` (Stop hook line 24) only runs when the parser branch is entered — but the parser is backgrounded after the cleanup, so even when it crashes the staging dir is already gone. **Re-reading the Stop hook, `rm -rf "$STAGING"` runs *before* the parser is backgrounded (line 24 vs. line 27).** That means a crashed parser leaves *no* staging dir to forensically inspect — the evidence is destroyed before the parser fails. This is worse than the candidate acknowledges. Cost at failure: zero forensic surface for the modal failure.
- **`hook-exec.sh` bug (action 2):** all three hooks fail simultaneously. Candidate accepts this under tradeoff #3 ("trade silent for noisy"). Operationally: the failure mode is at session-start (where one hook's failure means no `start.ts`, no `current-session-uuid`, no staging dir at all). Stop hook then has nothing to clean up; PostToolUse appends to a nonexistent path. Detection is loud (next `/explain` shows zero diagnostics across all sessions since the bug landed). Recovery requires inline-redirect restore *across three hook files*. ETA-to-recovery: 15–30 minutes if the restore commit is at hand; longer if not. The candidate names "minutes" implicitly; that is correct *only* if a versioned pre-wrapper snapshot exists. None is named.
- **Canary noise (action 6):** if the canary fires once during normal operation (e.g., the operator runs three sparse sessions in a row, all of which legitimately have low subagent activity because they're factual `canon-librarian`-only or `quick take` invocations), the warning surface erodes. The candidate's threshold ("3 consecutive sessions of `transcript-found: no` *or* `total-tokens: 0`") does not exclude bypass-mode invocations from the count. Bypass sessions structurally produce zero token-by-agent data because no subagents are invoked. **The canary will false-fire after three consecutive `quick take` invocations.** This is a routine operator pattern. Cost at false-fire: operator dismisses the warning; the next *real* failure is dismissed by reflex.
- **Replay experiment crash (action 3):** the candidate does not name what happens if one of the three replay sessions errors out. Outside-view's gap-list flagged that pre-handoff sessions have no `workflow-id.txt`, so the parser may be invoked with arguments it cannot resolve. If action 3 cannot run, actions 1, 4, 5 are still viable; action 6 (canary) was *informed* by action 3's findings and would ship without that input.

## 6. Frame-level objection

The candidate frames diagnostics as a **measurement layer** (which fields to keep, which to kill, which to defer). The operational view is that this is a **two-process synchronization problem with no synchronization primitive**. The Stop hook backgrounds a parser; the orchestrator's step-13 ledger-render runs at an undefined offset relative to that background process; consumers fall back silently when the race loses. The candidate's six actions all live inside the measurement frame — they prune fields, document schemas, name exit criteria. None of them addresses the synchronization gap that determines whether *any* measurement reaches a consumer. Action 6's canary detects the gap *after* it has already corrupted three ledger entries; it does not close the gap.

A concrete reframe: instead of "which metrics survive?" ask "what is the synchronization contract between the Stop hook's parser and ledger-render?" Options include (a) make the parser synchronous in the Stop hook (cost: hook latency increases by parser runtime, currently <100ms claimed but unvalidated per Explore gap-list), (b) make ledger-render explicitly wait-and-retry for `metrics.json` with a bounded timeout, (c) make the parser write a `parser.lock` sigil that ledger-render polls, (d) accept the race and explicitly mark sessions where the race lost so silent-skip becomes loud-skip. The candidate picks none of these and does not name the choice as a choice.

A second frame-level objection: the candidate accepts staging-dir bloat (orphan UUIDs from pre-handoff sessions, plus any future session where the orchestrator forgets the step-1 workflow-id write) without proposing retention. Convention-only cleanup is the same anti-pattern the candidate's action 2 calls out for `2>/dev/null`. The argument that structural enforcement matters for AI-blindness applies symmetrically to staging retention: a future maintainer running 100 sessions without the handoff produces 100 orphan dirs that nothing prunes.

## 7. Verdict

**rework**

Specific changes that would change the verdict to `approve`:

1. Action 2 must specify an additive rollout (wrapper introduced alongside inline redirects, n-session overlap, then cutover) and name the rollback artifact (a tagged commit or a `pre-wrapper-snapshot/` dir under the upgrade entry).
2. Action 6's canary must (a) exclude `quick take` and bypass sessions from the consecutive-failure count, or use a denominator-aware rule like "3 of last 5 non-bypass sessions" rather than "3 consecutive any-sessions," and (b) emit a sigil distinguishing parser-crashed from parser-ran-empty so the silent-skip and silent-zero-attribution failure modes are not collapsed.
3. The candidate must address the Stop-hook synchronization race directly — name the choice (synchronous parser, wait-and-retry, lock sigil, or loud-skip) and accept its cost — rather than relying on ledger-render's silent-skip as the sole control. "Silent skip" is a fallback, not a contract.
4. Add a staging/orphan retention policy (named retention window, named pruning trigger) so the directories named in Explore §11 do not accumulate indefinitely.

What I would watch if this shipped today (despite the rework): the count of completed workflow sessions where `<target>/.claude/session-artifacts/<id>/diagnostics/metrics.json` is absent at the moment `synthesis.md` is written; the count of `<target>/.claude/.metrics/staging/` subdirs not removed within 24h of their `end.ts`; and whether action 5's Heilmeier exam is reviewed at all by month six.
