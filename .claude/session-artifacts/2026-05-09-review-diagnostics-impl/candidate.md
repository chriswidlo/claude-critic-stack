# Candidate recommendation — review-diagnostics-impl

## Position (one sentence)

**Treat the v1 diagnostics layer as overshipped: kill the metrics fields whose downstream workflow decision is unnamed, replace convention-only AI-blindness with a structural wrapper, name a Heilmeier exam, run the three-session replay experiment before any SOTA tightening, and convert the schema doc from "additive/optional" to a producer/consumer conformance contract.** This is a *kill-more-than-ships* posture, consistent with re-classifying the session as `extend` rather than `investigation` and with canon §As Simple as Possible.

## Frame-level objection from challenges.md (addressed first, not last)

The challenger's binding objection: *for each metric the diagnostics layer emits, name the workflow decision it changes; metrics with no named decision are candidates for removal under SRE §As Simple as Possible, not candidates for SOTA tightening.*

Walking the shipped `metrics.json` fields against that test:

| Field | Workflow decision it would change | Verdict |
|---|---|---|
| `duration_seconds` (total only) | None today. The `/explain` dashboard shows a stub "~10 min" average; that is a *vanity* render, not a decision-shaping signal. Per SRE §Worrying About Your Tail, single-mean aggregates obscure the failure modes the operator actually wants to debug. | **Demote** — keep the field collected (it's free), remove from the explain card and ledger render until per-step timing exists |
| `tokens.total` | Budget alarm at session level. Plausibly load-bearing once budgets are explicit; today no budget threshold is named, so the decision is latent. | **Keep, named-as-latent** — flag the missing threshold as a follow-up commitment, not a SOTA-style upgrade |
| `tokens.by_agent` | Directly answers "are critics balanced" / "did one agent dominate." Decision: re-prompt, re-weight, or retire a dominant agent. This is the load-bearing field. | **Keep** — this is the one metric whose decision-shape is clear today |
| `tool_calls` (counts) | Loose connection to debugging tool-thrashing agents. Cheap to collect, marginal to consume. | **Keep, marked decorative** — explicitly not a SOTA target |
| `verdicts` (parsed) | None. The source of truth is `critiques/*.md` and `decision-log.md`; the parsed copy in `metrics.json` is *derivative*. Worse, its regex is loose (verdict regex no `\b` prefix; explore §2 line 182). The copy adds maintenance + false-positive surface for no incremental decision. | **Kill** — remove from the parser. Consumers (ledger-render) read `critiques/*.md` directly if they need verdicts |
| `loops` (parsed) | Already capped at 2 by CLAUDE.md step 11. Cross-session "are we loop-thrashing" is the only marginal use. Current regex grep across `decision-log.md` prose is fragile. | **Tighten or kill** — replace `\b(rewrite\|replan)\b` grep with `## Loop N` header count; if that's not cheap, drop the field |

The challenger called for "the candidate must explicitly list which (if any) shipped fields fail this test rather than treating all of them as load-bearing by default." Two fields fail (`verdicts`, `duration_seconds`-as-rendered) and one is conditionally fragile (`loops`). The remaining three pass. Net: the layer is over-instrumented for the decisions it actually drives.

## Concrete actions, ranked

1. **Kill the `verdicts` parsing path.** Delete the verdict regex (`<target>/bin/parse-session-metrics.py:170-185`); remove the `verdicts` field from `metrics.json` schema; update ledger-render and explain to never read it. **Why first:** smallest, cheapest action; removes a named correctness defect (loose regex, false positives flagged in upgrade entry); consistent with "source of truth is the critiques/ files." Cost: ~30 min.

2. **Replace the `2>/dev/null` convention with a `<target>/.claude/hooks/hook-exec.sh` wrapper.** Wrapper takes a command, runs it under `set +e` with stdout+stderr both redirected to `/dev/null` (or to a staging log), and is the *only* path the three hooks use to invoke external commands. A future maintainer who adds `echo "DEBUG: ..."` to a hook still cannot leak because the wrapper enforces redirection structurally. **Why second:** this is the directly-named below-base-rate feature in the outside-view distillation; converts a convention-enforced invariant into a structurally-enforced one; bounded surface (~30 lines new, three hook scripts edited). Cost: ~1 hour.

3. **Run the outside-view's cheapest experiment.** Replay three pre-2026-05-09 session-artifact directories' transcript JSONL files through the parser in isolation: `2026-04-27-critics-get-write-tool-impl` (complete workflow), `2026-04-25-sota-claude-online-research`, and one more. For each: does it produce `metrics.json`? Do `tokens.total` and `tokens.by_agent` sums match within 5%? Does `loops` count match a hand-count of `## Loop N` headers in `decision-log.md`? **Why third:** breaks the dogfooding sample-of-one; stresses the undocumented-layout dependency on samples the implementation has never seen; cheap (~1 hour). Concrete falsification for outside-view's "below base rate" call. Cost: ~1 hour.

4. **Extend the schema doc with a producer/consumer conformance section.** The challenger's row-6 objection is binding: the doc treats `## Metrics` as additive/optional but specifies *no* contract on what fields consumers (ledger-render, explain) are entitled to assume. Add a "Metrics conformance" section to `<target>/.claude/session-artifacts/README.md` declaring: required fields (`tokens.by_agent`, `tokens.total`); optional fields (`tool_calls`, `loops`); unit/type for each; consumer fallback rule (silent skip if missing-required, never error). **Why fourth:** prevents the deferred-problem failure mode the challenger named (parser produces stale-schema `metrics.json` after host layout change, consumers render confidently against malformed input). Cost: ~30 min.

5. **Name a Heilmeier exam.** Add to the upgrade entry's lifecycle row: *value-proved condition* — "at least one orchestrator step (a CLAUDE.md edit, a skill change, a session pattern, an agent prompt change) is committed in response to a `metrics.json` reading within 6 months." If 6 months elapse with no such commit, the layer enters a removal review. **Why fifth:** addresses the canon Heilmeier objection at the frame level; converts the layer from "instrumentation that justifies its existence" to "instrumentation with a named exit criterion." Cost: ~10 min.

6. **Add a parser canary.** Periodically (or at every Stop), the parser writes a `diagnostics/canary.txt` containing: `transcript-found: yes/no`, `subagent-count: N`, `total-tokens: N`. If three consecutive sessions report `transcript-found: no` or `total-tokens: 0`, the next `/explain` invocation surfaces a one-line warning: "diagnostics parser has produced no data in the last 3 sessions; check `<target>/bin/parse-session-metrics.py` against current Claude Code transcript layout." **Why sixth:** converts the modal failure (silent attribution drift) from silent to noisy. Cost: ~1 hour.

7. **Defer everything in the upgrade entry's SOTA list (per-step duration, cost-in-$, prompt/response length distributions, cross-session aggregation, privacy mode).** None of these have a named workflow decision they would change today. Per canon §As Simple as Possible, signals "rarely exercised … should be up for removal" — and these are not collected, much less rarely exercised. They re-enter consideration only after (a) actions 1–6 have landed, and (b) the Heilmeier exam in action 5 has been met at least once. **Why deferred, not killed:** they are not artifacts that exist; they are wishlist items. Killing wishlist items is cheap; deferring them with a named re-entry condition (action 5's exam) is more useful.

Total cost of actions 1–6: ~4 hours. Total *removed* surface: a regex, a field, a fragile invariant, two phantom future commitments. Total *added* surface: a 30-line wrapper, a conformance section in the schema doc, a canary file, a one-line success exam.

## Named tradeoffs

- **Vs. "ship the SOTA list."** This recommendation rejects the upgrade entry's SOTA section as the panel's agenda. Tradeoff: the operator's catalyst question "where did the 10 minutes go?" remains *unanswered* in v1.x — per-step timing is deferred, not built. Consequence: the dashboard's per-step bucket is empty until action 5's exam is met. The user must accept that "minimally canonical" (per SRE four-golden-signals) is acceptable for v1, and that "tail-aware" (per SRE §Worrying About Your Tail) is a v2 question gated on usage evidence.
- **Vs. "leave verdicts in metrics.json."** Killing the `verdicts` field forces ledger-render and explain to read `critiques/*.md` directly when they need verdict data. Tradeoff: small additional work in those skills (already implicit in the design — they had access). Cost is real but low.
- **Vs. "trust the convention."** Action 2 (`hook-exec.sh` wrapper) is ~1 hour of work for a low-probability, high-blast-radius failure mode (an AI-visible hook leak corrupts a critic's verdict). The tradeoff is that the wrapper adds an indirection layer; a maintainer reading `session-start.sh` now reads "call hook-exec.sh date" instead of "date 2>/dev/null". Marginal cost; structural enforcement is worth it.
- **Vs. "skip the replay experiment."** Action 3 is ~1 hour but produces zero artifacts the operator will use again. The output is a *yes/no* on whether the parser handles non-dogfood transcripts. Tradeoff: it could be skipped, and we'd be asserting "parser works on real workloads" without evidence. The replay is unusual in that its *value is in failing*: if the parser succeeds, the experiment costs an hour for no learning; if it fails, it surfaces the modal failure mode now rather than at month six. Asymmetric value-of-information justifies the cost.
- **Vs. labeling the session `extend` instead of `investigation`.** The challenger argued this label flip would shift the panel from "queue-the-backlog" to "kill-or-ship." This recommendation already adopts the kill-or-ship posture; the label is downstream of the posture. Tradeoff: leaving the classifier label as `investigation` while taking `extend`-shaped actions is mildly inconsistent. Synthesis should declare the slip explicitly rather than hide it.

## Named assumptions (≥ 3 that would flip the recommendation if wrong)

1. **The operator runs <20 workflow sessions in the next 6 months.** If true, averages and cross-session aggregations are pre-empirical (the Anthropic ≥20 baseline in canon distillation), and the SOTA wishlist is queued against a dataset that may never reach statistical relevance — the kill-or-defer posture is correct. *Flip condition*: if the operator runs ≥20 sessions in 6 weeks, the data volume justifies tightening per-step timing and cross-session aggregation now, and the deferral in action 7 becomes premature.
2. **Per-step timing is harder to ship correctly than v1 implies.** This recommendation defers per-step timing on the assumption that doing it right means clustering events around artifact mtimes (per upgrade entry's own SOTA section), which has fragile boundary conditions (what is "step 5" when steps 3-5 run in parallel?), not just additional log lines. *Flip condition*: if per-step timing turns out to be a 1-hour change (e.g., the orchestrator already emits a step-marker file each step), action 7's deferral inverts and per-step timing should land alongside actions 1–6.
3. **The undocumented Claude Code transcript layout will change in 2026.** Outside-view's modal failure assumes Claude Code's release cadence makes a 6–9 month layout shift likely. *Flip condition*: if Anthropic publishes a stable hooks-output spec or transcript-format SLA, the parser's coupling to `~/.claude/projects/<encoded-cwd>/<uuid>.jsonl` becomes loose-coupling-acceptable, the canary in action 6 becomes lower-value, and the canon SRE objection (loose-coupling violation) substantially weakens. Action 6 stays useful as defense-in-depth but no longer load-bearing.
4. **The `/explain` dashboard's primary value is the orientation card, not the metrics averages.** The current dashboard renders "~10 min" because <3 sessions have populated `duration_seconds`. This recommendation treats that "~10 min" as a vanity render, not a load-bearing operator-facing claim. *Flip condition*: if the operator routinely uses the explain card's averages to make scheduling or budget decisions ("can I run a session before lunch?"), demoting `duration_seconds` from the explain card breaks that workflow and the action-1-table verdict on `duration_seconds` is wrong.
5. **No critic verdict has ever been corrupted by a stderr leak.** Action 2 (`hook-exec.sh` wrapper) treats AI-blindness as load-bearing because a leak would corrupt critic decisions. *Flip condition*: if review of past sessions confirms zero historical leak (e.g., grepping the existing transcript JSONL files for the strings `set +e` or `2>/dev/null` shows no hook output ever entered an AI context), the structural-enforcement priority drops; convention-enforcement might be sufficient and the action could be deferred.

## Named ways this could be wrong

- **The kill-the-verdicts-field decision could be premature.** The parsed verdicts may turn out to be cheap input to a future cross-session "verdict pattern by lens" dashboard — the kind of analysis that, if a critic lens systematically rejects in 80% of sessions, would actually point to a frame mismatch. Killing the field today closes that door and forces a re-extract from `critiques/*.md`. The recommendation accepts this risk because the source-of-truth migration is cheap; if cross-session verdict analysis becomes a live need, the field can come back with a tighter regex (or, better, structured frontmatter in `critiques/*.md` that the parser pulls out by key, not by regex).
- **The `hook-exec.sh` wrapper could itself become a single point of failure.** If the wrapper has a bug, *all three* hooks fail, not one. The current convention-based discipline has independent failure points per hook line. The recommendation accepts this concentration risk because a wrapper-bug is a noisy failure (something obviously breaks across all sessions immediately), while a convention-violation is a silent failure (one hook leaks, one critic verdict is corrupted, nobody notices). Trade silent for noisy.
- **The Heilmeier exam (action 5) could harden into theater.** If "at least one orchestrator step is committed in response to a metrics reading" becomes a checkbox, the operator could ship a token-budget threshold that nobody acts on and call the exam met. The recommendation does not specify how to keep the exam honest; that is itself a deferred problem.
- **Action 7 (defer the SOTA list) might be over-correction.** Per-step timing has the strongest claim of the SOTA items — it's the gap that the catalyst question opened with. Deferring it for 6 months of usage evidence may be the right answer or may be canon-applied-too-aggressively. If action 3 (replay experiment) shows the parser works cleanly on non-dogfood data, the operator may want to spend the resulting confidence on per-step timing rather than wait for the exam.
- **The session label probably should have been `extend`.** Per the challenger's classifier critique, the scope-map already produces an extend-shaped output. Continuing under `investigation` while taking extend-shaped actions risks documentation drift between the label and the work. Synthesis should declare and resolve this.

## What success looks like (for synthesis)

A panel-approved version of this recommendation produces, in roughly the next session of work:

- Two files deleted/shrunk (`bin/parse-session-metrics.py` loses ~15 lines for the `verdicts` parser; `metrics.json` schema in the README loses one field).
- One file added (`<target>/.claude/hooks/hook-exec.sh`).
- Two files edited (`<target>/.claude/session-artifacts/README.md` for the conformance section; `<target>/upgrades/normal/2026-05-09-ledger-records-time-tools-tokens/README.md` for the Heilmeier exam line).
- One experiment run (parser replay against three named pre-2026-05-09 sessions, results recorded in this session's `decision-log.md` or a separate `experiments/` dir).
- One follow-up upgrade entry queued (`upgrades/normal/<date>-diagnostics-canary-and-per-step-timing/`) with a re-entry condition tied to the Heilmeier exam.

Concrete, scoped, reversible. Six months from now, the diagnostics layer is either a smaller, harder, more-load-bearing artifact or a documented removal. No middle path through the research-theater bias.
