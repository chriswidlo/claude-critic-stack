# Candidate v2 — `upgrades/` lab design (post-replan, post-frame-Revision-2)

Session: `2026-04-26-upgrades-lab-design`. Step 9, second pass. Replaces the v1 candidate that received three `rework` verdicts at Step 10. The v1 candidate is preserved in git history; this v2 is generated against frame.md Revision 2.

## Position

Design `upgrades/` as a **thin capability-overlay on existing session-artifact promotion + a capture-first inbox surface**, with exactly **two locking primitives** (slug-based stable IDs + the promote operation). All other apparent structure — document type set, maturity ladder, cross-reference vocabulary, disciplines, subfolder organization — is **additive metadata over the primitives** that can be refactored without rewriting accumulated content. Three operational controls (ratio-script, break-glass protocol, curator-bootstrap path) are required outputs of this candidate, not §Tradeoffs concessions. The directory name `upgrades/` is preserved (per operator's earlier choice) with a deferred decision on rename to `lab/` flagged as a Step-12 uncertainty, not a Step-9 position.

This candidate's load-bearing claim: **adding a new document type, renaming the maturity ladder, reorganizing disciplines, or restructuring subfolders does not require touching any prior entry's content.** That is the test of the frame, demonstrated below.

## Locking primitive 1 — Slug-based stable IDs

Every entry has a slug assigned at creation: `<YYYY-MM-DD>-<short-kebab>` (matches existing session-artifact convention). The slug is the entry's logical identity. Cross-references between entries use slugs, not filesystem paths.

```yaml
# in any entry's frontmatter
slug: 2026-04-26-lab-design
relates_to: [2026-04-25-shadow-comparator]
```

`upgrades/INDEX.md` is the slug → current-path map, **derived from on-disk frontmatter** (not hand-edited; regenerated by `bin/upgrades-index.sh`). The map is a cache; the source of truth is each entry's frontmatter `slug:` field. Renaming a file, moving entries into subfolders, or restructuring the directory updates only the path; cross-references survive.

**Why this locks:** the slug scheme is referenced by every cross-reference in every entry. Changing it (e.g., from `<date>-<kebab>` to `<uuid>`) would require rewriting every entry's `relates_to` field. So slug syntax is locked. The mapping mechanism (INDEX.md derivation script) is reversible — it can be replaced by a database, a graph store, or any other index without touching entries.

**What this primitive replaces from v1:** four locking dimensions (document types, maturity ladder, cross-ref vocabulary, integration protocol). All four are now additive frontmatter over the slug primitive.

## Locking primitive 2 — The promote operation

A single operation: take any artifact (a session-artifact directory, a `pages/` note, an inbox capture, a `plans/` deferred item) → assign a slug → add a minimal frontmatter (`slug`, `created`, `type`, `maturity`) → move or hard-link into `upgrades/` → append a one-line entry to `INDEX.md`.

The operation is implemented as `bin/upgrades-promote.sh` (a thin wrapper script, not an agent — agents are the future expansion path). Initial implementation: ~30 lines of bash. Takes source path + slug. Outputs the new path in `upgrades/`.

```bash
./bin/upgrades-promote.sh .claude/session-artifacts/2026-04-25-shadow-comparator/synthesis.md 2026-04-25-shadow-comparator
```

**Why this locks:** every entry in `upgrades/` is reached by this operation. Changing the operation's contract (e.g., requiring more frontmatter at promote time) is rewrite-forcing if existing entries don't have the new fields. So the *minimum required frontmatter at promote time* is locked. Adding optional frontmatter (any field beyond the four named) is additive and reversible.

The four required fields, justified individually:

- **`slug`** — needed by primitive 1 (stable IDs).
- **`created`** — date of the source artifact (or capture for inbox entries). Needed for chronological queries; non-locking change to the format is rewriteable since dates are well-formed.
- **`type`** — a tag value from a starter set (see below). Required so queries can filter; non-locking value-set change is additive.
- **`maturity`** — a tag value from a starter set (see below). Required for the same reason.

## Capture-first surface — `upgrades/inbox/`

Express path. Entries in `upgrades/inbox/` have only `slug`, `created`, and a body. No `type` (defaults to `observation`), no `maturity` (defaults to `raw`), no `relates_to`.

The operator captures a thought in <30 seconds: create `upgrades/inbox/<YYYY-MM-DD>-<slug>.md`, write whatever, save. Done. The entry has been "processed" in the philosophy-of-the-stack sense (a slug was assigned; the artifact has landed) but has not been "structured."

A separate promotion pass — operator review or curator agent (whichever exists) — promotes inbox entries into the typed/matured structure when they earn it. The pass moves entries from `upgrades/inbox/` to `upgrades/` proper, adding `type` and `maturity` frontmatter.

This dissolves the v1 candidate's lab-vs-backlog tension: inbox entries are *captured* but not *triaged*; the no-untriaged rule is preserved by treating capture itself as a maturity rung implicit in the inbox path.

## Additive metadata (reversible, view-shaped)

Everything below this line is **starter tag-sets**. They can be extended, renamed, or restructured without rewriting any prior entry's body. Renaming requires only updating the `lint` script (below) and any entries authored afterward — prior entries with the old tags continue to be valid; the lint accepts both old and new for a transition period.

### Starter document type tags (4)

- `observation` — surfaced pattern, signal, or gap. No claim.
- `claim` — testable assertion. May be a hypothesis, position-paper, or RFC at any maturity.
- `decision` — recorded implemented decision (the ADR-shaped artifact).
- `killed` — explicit rejection with reason.

This is half the v1 set. New types (`hypothesis` distinguishing from `position-paper`, `rfc` distinguishing from `claim`, `experiment`) can be added when the corpus reveals the distinction is doing work. The candidate's load-bearing claim is that this addition does not require rewriting existing entries — they keep their `claim` tag; only newly-authored entries use the finer types.

### Starter maturity tags (2)

- `raw` — captured, possibly inboxed, no validation.
- `matured` — promoted out of inbox; has prior-art check or evidence per type-specific criteria documented in `upgrades/PROMOTION.md`.

This is binary. Finer ladder (`articulated`, `tested`, `integrated`) can be layered on later as additive states without rewriting; existing `matured` entries can be sub-classified if the operator chooses.

### Starter discipline tags

Inherit from `MODULES.md` (which is *also* to-be-written; it owns the seven-capability taxonomy per scope-map conflict #12). Until `MODULES.md` exists, entries use `discipline: cross-cutting` as a placeholder. Once `MODULES.md` lands, entries can be re-tagged in a single batch operation; the lint script enforces tag validity post-`MODULES.md`.

### Starter cross-reference primitive (1)

```yaml
relates_to: [<slug>]                       # bare form, sufficient for most uses
relates_to:                                # rich form, optional
  - slug: <slug>
    relation: builds_on | refutes | supersedes
    reason: <one short sentence>
```

The `relation:` tag is optional. New relation values can be added without rewriting prior entries. The bare form is the default; the rich form is used when the relationship is doing semantic work (not "they're related," but "this refutes that with reason X").

## Three required operational controls

### Control 1 — Ratio-script (instrumented falsifier)

`bin/upgrades-ratio.sh`. Counts git commits since session synthesis date.

- **Meta-structure commits:** commits touching `upgrades/INDEX.md`, `upgrades/PROMOTION.md`, `upgrades/SCHEMA.md`, `upgrades/TIMEBOX.md`, `bin/upgrades-*.sh`, or any markdown adjacent to the lab structure (not entries themselves).
- **Content commits:** commits creating or substantially editing files under `upgrades/` *except* the meta-files above. Inbox captures count as content.
- **Output:** ratio `meta:content` and the absolute counts. Optionally, a deny-list classification per commit (printed when `--debug` is passed).

Run on demand by the operator. A pre-commit hook (optional, default-off; behind env var `UPGRADES_RATIO_HOOK=1`) prints the current ratio after each commit. Day-60 trigger: when ratio exceeds 3:1 by elapsed-day-60-from-synthesis, the operator is prompted (in the hook output if hook is on; via manual run if not) to: (a) note the trigger, (b) write a `killed` entry naming what structural commitment was wrong, (c) freeze meta-commits until ratio recovers below 2:1.

Time cost to write the script: ~1 hour. Placed on the day-7 critical path.

### Control 2 — Break-glass protocol

In-quarter revision of any locking primitive (slug scheme or promote operation):

- **Trigger:** operator identifies a locking primitive is wrong before quarterly review.
- **Action:** write a `killed` entry in `upgrades/` naming what was wrong + the new commitment. The killed entry uses the *old* primitive (so the audit trail is preserved). The next entry uses the *new* primitive.
- **Cost:** resets the day-60 audit clock to "the day after the killed entry was written." Bounds the operator to one break-glass per quarter.
- **Recovery signal:** ratio recomputed against the new clock. Same threshold (3:1).

This makes the freeze actually freezable but not nominal — break-glass costs something and is bounded.

### Control 3 — Curator-bootstrap path: lint script

`bin/upgrades-lint.sh`. ~50 lines. Validates each entry under `upgrades/`:

- Frontmatter contains required fields (`slug`, `created`, and (if not in inbox) `type`, `maturity`).
- `slug` is unique across the corpus.
- `slug` value matches the filename (when not in inbox).
- `type` value is in the active starter set (or extended set as documented in `upgrades/SCHEMA.md`).
- `maturity` value is in the active set.
- `relates_to` slugs all resolve to existing entries (or to the entry itself, allowed).
- `discipline` (when present) is in the active set per `MODULES.md` (when `MODULES.md` exists).

Run on demand: `./bin/upgrades-lint.sh`. Optionally as a pre-commit hook (off by default). The full curator agent (item E in the initial backlog) is *not* a day-7 dependency — it is a future capability that may layer on this lint as its validation primitive.

This addresses the operations critique's chicken-and-egg: schema validation does not require an agent; a script suffices. Manual writes are validated by the script; the curator agent, when built, will use the same script.

## Demonstration that the additive layer is genuinely additive

The frame's load-bearing claim was: *adding a new document type, renaming the maturity ladder, reorganizing disciplines, or restructuring subfolders does not require touching any prior entry's content.*

Worked example. Suppose on day 100 the operator wants to add a fifth type, `experiment`, to distinguish entries that report on a test from entries that make a claim. Steps:

1. Update `upgrades/SCHEMA.md` to add `experiment` to the type set.
2. Update `bin/upgrades-lint.sh` to accept `experiment` in addition to existing types.
3. Newly authored entries can use `experiment`. **Existing entries with `type: claim` are still valid.** No body or frontmatter rewrite of any existing entry.

Worked example. Suppose on day 200 the operator wants to rename the `matured` state to `validated`. Steps:

1. Add `validated` as a synonym to `matured` in `upgrades/SCHEMA.md`.
2. Update `bin/upgrades-lint.sh` to accept either.
3. Newly authored entries use `validated`. **Existing entries with `maturity: matured` are still valid.**
4. Optional: a one-shot rename pass (`sed` over frontmatter) is available but not required. Either choice is honest.

Worked example. Suppose on day 300 the operator wants to subfolder by discipline:

1. Move entries into `upgrades/<discipline>/`. Filenames change.
2. `bin/upgrades-index.sh` regenerates `INDEX.md`. Slugs unchanged; cross-references survive.
3. Path-references-in-prose-bodies (rare) need manual update; slug-based `relates_to` references are unaffected.

If any of these worked examples requires more than the steps shown, the candidate is wrong — and day 7 ratification is the moment to find out.

## Frame-level objection addressed

The convergent panel objection from v1 — *the lab over-builds new structure when capability-overlay + capture-first frame would cut the locking-dimension count from 4 to 1–2* — is addressed by:

1. **Two locking primitives, both narrow:** slug scheme + promote operation. Down from four.
2. **Capture-first surface (`inbox/`)** as the operator's express path: <30-second capture with no metadata required.
3. **Everything else as additive metadata:** type tags, maturity tags, cross-references, disciplines, subfolders all reversible without entry rewrites — demonstrated by three worked examples.
4. **Three operational controls instrumented:** ratio-script, break-glass protocol, lint-script (curator-bootstrap stand-in). All four scripts (counting `upgrades-promote.sh`) are ~3 hours of bash; on the day-7 critical path.
5. **Operator's name preference preserved:** `upgrades/` retained; `lab/` rename deferred to Step-12 uncertainty for operator decision.
6. **Existing primitives subsumed honestly:** session-artifact promotion is the lab's actual generic, per architecture critique. The promote operation makes this explicit. `exemplars/` becomes a special case of promotion (matured exemplars; inbox is `raw`).

## Tradeoffs (named)

- **Two starter document types and binary maturity is *intentionally* under-specified.** If the operator wants more typology immediately, they can extend the starter set on day 7 — but the candidate's claim is that under-specification at day 7 is a feature, not a bug. The day-60 audit checks whether the operator *needed* the extension or whether the under-spec held.

- **The promote-operation is bash, not an agent.** This is a deliberate downgrade from v1's curator-as-monopoly. Bash scripts have lower expressiveness, no in-loop reasoning. Mitigation: the operations critique flagged the agent as the wrong primary dependency; bash is the right primitive at this scale; an agent can be built later as a layer over the script.

- **`upgrades/INDEX.md` as a derived cache means entries' `slug:` field is the source of truth.** If frontmatter is malformed, INDEX may be incomplete. The lint-script catches malformed frontmatter; without lint enabled, INDEX could silently miss entries. Mitigation: lint is documented as a pre-commit hook the operator should enable; the candidate names the manual fallback (`./bin/upgrades-lint.sh` before each commit).

- **The break-glass costs one quarterly review per use, but the candidate does not specify what happens if a *second* break-glass is needed in the same quarter.** Decision: second break-glass in a quarter is forbidden; the operator must wait. This is a real constraint; named explicitly here so the operator knows.

- **The candidate keeps the directory name `upgrades/` against my own (and the panel's) read of its semantic.** Reasoning: the operator's explicit naming preference outweighs the semantic concern at this stage; the rename is presented as a Step-12 uncertainty, not a Step-9 imposition.

## Assumptions that flip the recommendation if wrong (≥3)

1. **The slug primitive survives.** If the operator finds slug-based cross-references too cumbersome (e.g., needs to copy slugs by hand often, gets typos), the primitive must change. Mitigation: lint-script catches slug typos; if friction is high anyway, switch to UUIDs (rewrites references but preserves entries).

2. **The two-state binary maturity (`raw | matured`) is operationally usable.** If the operator finds binary insufficient on day 30 (e.g., they want to mark an entry as "tested but not yet integrated"), the additive-states claim is tested. Mitigation: extend the maturity set in `SCHEMA.md`; existing entries unchanged.

3. **The four-script tooling (~3 hours of bash) is actually written by day 7.** If the scripts are not written, the locking primitives are not actually instrumented, and the operator falls back to manual practice with no audit. Mitigation: the day-7 freeze does *not* land if the four scripts are not on disk; the freeze itself is contingent on instrumentation.

4. **The capture-first inbox is used.** If the operator continues writing in `pages/` instead of `upgrades/inbox/`, the lab fails by Mode A's silent-decay variant. Mitigation: the day-60 audit notices this (low entry count); the recovery is to ask why the operator preferred `pages/` and adjust friction.

5. **Self-application holds.** If the candidate cannot be promoted into `upgrades/` as its own first entry under the proposed primitives without contortions, the primitives are wrong. The day-7 ratification is the test.

6. **Operations controls are actually run.** A ratio-script that exists but is never run is theatre under a different name. Mitigation: the pre-commit hook makes the script self-executing; the candidate names the hook as the recommended-on configuration.
