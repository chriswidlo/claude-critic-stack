# TDD, Spec-Driven Development, and Verification-Before-Completion — 2026 Reference

> A canonical, stack-neutral reference on the engineering-discipline axis of AI workflows: how to specify what you want before generation, how to write tests that survive contact with an agent, and how to verify what the agent claims it did. For any practitioner. Not anchored to any stack.

Scope: software engineers, ML practitioners, and platform teams shipping with agentic coding tools (Claude Code, Cursor, Codex, Aider, in-house harnesses). Where a concrete artifact is named (Anthropic's TDD prompt sequence, Kiro's `requirements.md`, Spec-Kit's `/specify` command, Stryker, Hypothesis), it is a case study, not a prescription.

---

## 1. Frame: why this matters more in 2026

The bottleneck has moved. In 2023 it was generation — could the model produce a coherent function? By 2025 it could. By 2026 the constraint is *verification*: can you tell, cheaply and reliably, whether the coherent function it produced actually does what you asked?

The GitHub Engineering Blog gave the failure mode a name in early 2026: **vibe-coding**. A developer describes what they want in prose. The agent generates a plausible block. It compiles. The linter is clean. It looks right. Two days later, in production, it subtly misses intent — an edge case unhandled, a boundary off by one, a side effect skipped, an integration mocked when it should have been real. The test suite, if it existed, was generated by the same agent that wrote the implementation, and it asserted on the *output the implementation produced* rather than the *behavior the requirement demanded*.

The countermove has converged from three directions under three names — **TDD-with-agents**, **spec-driven development (SDD)**, and **verification-before-completion** — but the underlying shape is the same: *make the success criterion machine-checkable before generation; re-run the checker after.* The criterion can be a failing test, a structured spec, a JSON schema, a screenshot diff, a golden-dataset similarity score, or a mutation-testing report. In every case the discipline is the same: agents narrate intent and completion interchangeably; only deterministic evidence distinguishes them.

## 2. Test-driven development with AI agents

### 2.1 The classic loop, adapted

Kent Beck's red-green-refactor cycle survives the agent transition almost unchanged. What changes is which steps are non-obvious and which become load-bearing in a way they weren't for a human pair-programmer. The 2026 six-step version:

1. **State the desired behaviour in prose.** Not "add validation"; "reject any email field whose local part exceeds 64 octets per RFC 5321, returning a 422 with `error.code = "email_too_long"`."
2. **Agent (or human) writes a failing test.** The test encodes the prose criterion as an executable assertion.
3. **Watch the test fail for the expected reason.** Not just "red"; *red for the reason you predicted*. A test that fails because of an import error is not yet a test.
4. **Commit the failing test as a checkpoint.** This is the non-obvious step and it is load-bearing — see §2.3.
5. **Agent generates the implementation, loops until the test passes.** This is where the agent's speed advantage lives.
6. **Diff the test file against the checkpoint.** If the agent edited the tests, revert and re-prompt. The test is the spec; the spec does not get to change because the implementation found it inconvenient.

Step 3 is the one humans skip and agents punish. A passing test that was never observed failing tests nothing — it might pass against any implementation, including the empty one. Step 6 is the one humans don't need because humans don't usually rewrite their own tests to make a bug go away; agents will, gladly, and silently.

### 2.2 Anthropic's published TDD guidance

Anthropic's *Claude Code best-practices* (2025) and the follow-up *Enabling Claude Code to work more autonomously* (2026) both single out test-driven development as **"the single strongest pattern for working with agentic coding tools, as each red-to-green cycle gives Claude unambiguous feedback."** The reasoning is structural: agents are reward-hackers by construction; a passing-test criterion is the cheapest reward signal that resists hacking, *provided* the test isn't editable mid-loop.

The recommended prompt sequence, near-verbatim from Anthropic's published examples:

> "Write tests for the auth module using pytest. TDD approach, no mock implementations."
>
> "Run the tests. They should all fail."
>
> "Commit the failing tests as a checkpoint."
>
> "Write the implementation. Do not modify the tests. Keep going until all tests pass."

Four prompts, four contracts, each defending a specific failure mode: *no mock implementations* (the agent stubs the interface and does nothing); *they should all fail* (the agent writes tests that already pass — tautologies, mirror-of-implementation); *commit the failing tests as a checkpoint* (the agent silently edits tests during implementation); *do not modify the tests* (restates the contract in the implementation prompt, where the temptation peaks).

The non-obvious instruction is the checkpoint commit. It is the *only* mechanism in the loop that makes test-tampering recoverable. Skip it and the agent's edit history blurs into the implementation diff; keep it and any post-hoc test change shows up as an isolated commit you can revert.

### 2.3 Tests as the operative specification

Once you accept that tests are the success criterion the agent optimizes against, tests *become* the operative specification — whatever else you wrote, in prose or in tickets, is decoration. This is true for human developers too, but agents make it true on a shorter time horizon: an agent doesn't read the ticket twice and reconsider; it reads the test once per loop iteration and converges.

Two consequences:

The superpowers `test-driven-development` skill formalizes the first as a rule: **"If you didn't watch the test fail, you don't know if it tests the right thing."** A test that has never failed is a hypothesis, not a verifier. Run it red before you run it green.

The second consequence is design discipline: **"write the wished-for API."** Before you write the function signature, write the test that calls the function the way you wish it could be called. The test then forces the signature into a shape the caller actually wants, rather than the shape the implementation found convenient. This works with or without an agent; it is amplified with one, because the agent will happily generate any signature you describe — the test is what constrains the description into something usable.

### 2.4 Three inversions of who writes what

The 2026 TDD-with-agents practice fragments along a single axis: who writes the test, and who writes the implementation. The three combinations have different risk profiles.

| Pattern | Test author | Impl author | Risk |
|---|---|---|---|
| Agent-both | Agent | Agent | Tautological tests; mirror-of-implementation bugs |
| Human-test, agent-impl | Human | Agent | Highest assurance; slowest |
| Agent-test, human-impl | Agent | Human | Useful for legacy coverage; risk = false coverage |

Agent-both is the default and the most dangerous. It produces the most code per unit time and the least signal per line of code. Without a separate verification primitive (mutation testing, property-based testing, code review of the test file specifically), it converges to a self-consistent fiction.

Human-test, agent-impl is what Anthropic's prompt sequence assumes when read strictly. The human writes the *first* failing test (or at minimum, the prose criterion the agent then encodes), and the agent implements against it. Slowest, highest assurance — the assurance comes precisely from the human being the one who decides what "correct" means.

Agent-test, human-impl is a legacy-code play. The agent writes characterization tests against existing behavior; the human then refactors or extends with those tests as a safety net. The risk is *false coverage* — the agent's tests assert on the current behavior including its bugs, freezing the bugs in place. Mitigation: treat the generated tests as a starting draft, not a finished suite, and run mutation testing against them before trusting them.

### 2.5 When AI-generated tests are a smell

Four recurring failure shapes for agent-authored tests, with field signals:

- **Tautology.** `expect(fn(x)).toBe(fn(x))`, or post-hoc assertions copied from observed output (`expect(result).toEqual({id: 7, created_at: "2026-04-19T…"})` where `id` is generated and `created_at` is `Date.now()`). The test always passes because both sides come from the same execution.
- **Mirror of implementation.** The test enumerates the same branches as the implementation in the same order. If the implementation forgot a case, the test forgot it too. The implementation and the test were written from the same flawed mental model; nothing in the loop ever stress-tested the model.
- **Mock-the-world.** Every collaborator mocked until what's being exercised is the mock, not the code. The agent reports "all tests pass"; the test never reached the network, the database, the filesystem, or the third-party API.
- **Snapshot of stochastic output.** Common in tests of LLM-using code: `expect(response).toMatchSnapshot()` is run against a non-deterministic call; the snapshot updates every run; the test never fails and never passes meaningfully.

The empirical signal: a jghiringhelli industry report (2026) on agent-authored tests found AI-reported 93% line coverage falling to 34% **effective coverage after mutation testing** — i.e., two thirds of the lines marked covered had no assertion that would catch a meaningful change to them. The gap is the size of the problem.

### 2.6 Property-based testing as the counter-pattern

Example-based tests assert on specific inputs; property-based tests assert on *invariants* that should hold across a generated distribution of inputs. The library generates inputs the developer wouldn't have thought of; the property is the spec.

Canonical 2026 libraries: **Hypothesis** (Python), **fast-check** (TypeScript/JavaScript), **PropEr** (Erlang), **ScalaCheck** (Scala). Hypothesis remains the reference implementation; fast-check has dominated the JS/TS ecosystem since 2024.

Two recent papers tie property-based testing directly to LLM workflows:

- arXiv **2506.18315** (*Property-Generated Solver*, 2025) reports **23.1%–37.3% relative pass@1 gains** over established TDD methods when the agent generates properties first, then implementation. The mechanism: properties are harder to overfit than examples, so the agent's loop converges toward correctness instead of mimicry.
- ACM **10.1145/3696630.3728702** (2025) reports that combining property-based and example-based tests catches **81.25% of injected bugs**, versus 68.75% for either alone. The combination dominates; the order does not matter much.

The 2026 practical recipe is small and durable: for each unit of behavior, write **one example test for documentation** (so a reader can see what a typical call looks like) and **one property test for coverage** (so the agent's loop optimizes against an invariant, not a memorized input/output pair). Both run in CI; the example test is the docs the property test lacks.

Property-based testing is also the cheapest defense against the tautology and mirror-of-implementation smells, because the inputs the property runs against are not chosen by the implementer.

## 3. Spec-driven development

If TDD makes the *test* the spec, SDD asks: what if we wrote the spec first, as an artifact, and then generated the test and the implementation against it? The 2026 toolkits answer in two different shapes, both shipped publicly within four months of each other.

### 3.1 Kiro (AWS) — the structured-document approach

**Kiro** launched in preview October 2025 and reached **GA in January 2026** as the successor to Amazon Q Developer, which AWS end-of-lifed at the same time. Kiro's thesis is that the spec is a document, the document has a fixed shape, and the agent's job is to fill the shape before writing any code.

Three documents are produced before any implementation, in order:

- **`requirements.md`** — user stories with **EARS-notation** acceptance criteria (see §3.4). Each story has triggers, responses, and exception conditions named explicitly.
- **`design.md`** — architecture overview, component decomposition, sequence diagrams, data flow. Generated from `requirements.md` and reviewed by a human before tasks are generated.
- **`tasks.md`** — a numbered implementation checklist, each task scoped small enough to be a single agent invocation. Generated from `design.md`.

The pricing signals where AWS believes the value sits: **$0.20 per spec-mode credit** versus **$0.04 per vibe-mode credit** — a 5× premium for the structured path. The market read is that AWS expects the spec-mode artifacts to amortize across many downstream agent invocations, while vibe-mode is per-shot disposable generation.

Kiro runs on ephemeral AWS accounts per task, which makes the side-effect verification problem (§4.3) a platform concern rather than the developer's.

### 3.2 GitHub Spec-Kit — the agent-agnostic approach

**GitHub Spec-Kit** (`github/spec-kit` on GitHub) launched September 2025 and reached **90k stars / 8k forks by May 2026**. Where Kiro is opinionated and vertically integrated, Spec-Kit is deliberately agent-agnostic — it ships **29 named coding-agent integrations** (Copilot, Claude Code, Cursor, Aider, Codex, Continue, Cline, and the rest of the 2026 coding-agent matrix) and lets the human pick.

The Spec-Kit workflow is three slash commands, run from any compatible agent:

- **`/specify`** — turns a prose prompt into a structured spec, similar shape to Kiro's `requirements.md` but unopinionated about the underlying schema.
- **`/plan`** — turns the spec into an implementation plan, agent-readable.
- **`/tasks`** — turns the plan into a numbered task list, one task per agent invocation.

The convergence between Kiro and Spec-Kit on three-document-first / numbered-tasks-second is not a coincidence — both teams independently landed on the same shape because it's the smallest one that survives the agent's narration-without-completion failure mode. A spec the agent must reference at task time keeps it grounded; a numbered task list keeps the agent's per-invocation scope small enough to verify.

### 3.3 Spec-as-code: formal methods

The high-end of the spec-driven spectrum is formal specification. **TLA+** (Lamport) and **Alloy** specify system invariants in a logic that a model checker can exhaustively search. **Lean** and **Coq** prove properties of generated code via dependent types and tactics.

The 2026 thesis on formal methods + LLMs is *not* that LLMs write Lean proofs — mostly they don't, and the proof assistants reject what they produce. The thesis is that LLMs can be **constrained by a formal spec**: the spec rejects any candidate implementation that violates the named invariants, regardless of how many examples the implementation passes. The agent loop becomes generate → check-against-spec → repeat, with the spec as an oracle the agent cannot edit.

Property-based testing is the cheap-and-popular instance of this same pattern (an invariant the agent is generated against). Formal methods are the expensive-and-correct extreme (an invariant *proven* to hold). Most 2026 production code lives on the PBT end of the spectrum; safety-critical, financial, and consensus-protocol code increasingly lives at the Lean/Coq end.

### 3.4 EARS notation

**EARS** ("Easy Approach to Requirements Syntax") is a closed-grammar way of writing acceptance criteria, originally from Alistair Mavin at Rolls-Royce (2009), re-discovered as load-bearing for SDD in 2025–2026. The canonical form:

> *the system shall [response] when [trigger]*

with variants for unwanted behaviour (*if [trigger] then the system shall [response]*), state-driven (*while [state] the system shall [response]*), and event-driven cases. The grammar is closed — fewer ambiguous constructions, no comma-spliced multi-requirement sentences.

Why it matters for agents: EARS sentences parse more reliably than freeform prose. An agent reading `the system shall return 422 with error.code = "email_too_long" when a registration request has an email field with a local part longer than 64 octets` extracts an unambiguous trigger and an unambiguous response. The same intent in freeform prose ("we should reject very long email addresses with a clear error") is interpretable in a dozen ways, and the agent will pick one.

Kiro defaults to EARS for `requirements.md`. Spec-Kit allows it without enforcing it. The 2026 consensus among SDD practitioners is that EARS or an EARS-like closed grammar is worth the friction at requirements time because it prevents a much larger ambiguity tax at implementation time.

### 3.5 README-driven development

Tom Preston-Werner's 2010 *README-Driven Development* post has been re-discovered as the lightweight end of SDD. The discipline: write the README first, before any code; the README is the spec; the tests are the verifier; the implementation is what gets generated from the spec. For projects too small to justify a `requirements.md` / `design.md` / `tasks.md` triplet, README-first is the irreducible minimum that still defends against the vibe-coding failure mode.

### 3.6 The arc most 2026 setups converge on

```
user story
   → acceptance criteria (EARS notation)
   → example + property tests
   → agent-generated implementation
   → diff + test verification
   → human acceptance
```

Each arrow is a checkpoint the agent cannot skip. Each artifact is something the agent reads but does not own — the human owns the user story, the EARS criteria, and the final acceptance; the agent owns the implementation; ownership of the tests is the axis from §2.4.

## 4. Verification primitives for AI output

The verification stack has layers. The minimum is unglamorous; the maximum is open-ended.

### 4.1 The minimum stack

Four checks, in order of cost and in order of necessity:

- **Build check.** Exit 0 from the build command. Necessary, not sufficient — a build can succeed against a program that does nothing.
- **Lint check.** Formatter and linter clean. Catches syntax-class hallucinations and obvious style misses; does not catch semantics.
- **Type check.** Full type-checker pass. The most under-rated single primitive in 2026 — type checking catches a huge fraction of API hallucinations, including the very common failure of agents calling functions that *almost* exist with arguments that *almost* match the real signature.
- **Test run.** The actual command. Not the cached results, not the previous run, not what the agent says happened. The superpowers `verification-before-completion` skill states the rule: **"If you haven't run the verification command in this message, you cannot claim it passes."**

The four together are necessary for any "I'm done" claim. None of them, individually or together, are sufficient — they are the floor, not the ceiling.

### 4.2 Frontend: visual iteration

Anthropic's 2026 visual-iteration pattern adds a fifth primitive specific to UI code. Claude Code (and equivalents in Cursor, Codex, etc.) starts a dev server, opens an embedded browser, takes screenshots, inspects the DOM, clicks elements, fills forms, and confirms behavior — auto-verifying after every edit.

The screenshot-driven loop:

1. Paste the target design (Figma export, Dribbble shot, hand sketch).
2. Agent generates the first implementation.
3. Agent screenshots the result and compares to target.
4. Loop until visual delta is acceptable.

This works because the verification primitive (visual diff) lives outside the agent's narrative. The agent can claim "I made the button match the design"; the screenshot says yes or no. The same shape — output-shape verification by something the agent doesn't author — runs through every primitive in this section.

### 4.3 Sandboxed execution for side effects

Per-task ephemeral environments are the 2026 default for any agent that touches the filesystem, the network, or external systems with state: **git worktrees** for filesystem isolation, **Docker** for runtime isolation, **ephemeral cloud accounts** (Kiro's approach on AWS) for full integration testing.

The point of these is *not* primarily security — it's reversibility. The verification step must be allowed to fail loudly without polluting the human's working tree, the dev database, or the third-party account. An agent that has nothing to lose verifies more honestly than an agent that knows a failed verification will leave the workspace dirty.

### 4.4 "Trust but verify" patterns: the agent-claim verification table

The superpowers `verification-before-completion` skill articulates the pattern as a table. Each row is an agent claim; the middle column is the evidence required *this turn*; the right column is what most humans accept and shouldn't.

| Agent claim | Required evidence | Insufficient |
|---|---|---|
| "Tests pass" | Test output: 0 failures, this turn | A previous run; "should pass"; "the changes look correct" |
| "Linter clean" | Linter output: 0 errors, this turn | Partial check; extrapolation from earlier output |
| "Build succeeds" | Build command: exit 0, this turn | "Linter passed"; "logs look good" |
| "Bug fixed" | Test of original symptom passes | Code changed, assumed fixed; reasoned about the cause |
| "Agent completed delegated task" | VCS diff shows expected changes | The sub-agent's own success report |
| "Requirements met" | Line-by-line checklist against the spec | Tests passing |

The discipline is the same in every row: the evidence must come from a deterministic system, the evidence must be from *this* turn, and the agent's narration alone is never enough.

### 4.5 Diff-based verification

The single most useful 2026 verification habit, stated as a one-liner: **after any agent says "I did X," run `git diff` and confirm X is in it.**

Agents narrate intended actions and completed actions interchangeably. "I'll update the config file and add a test" can mean any of: I'm about to do both, I did both, I did one and forgot the other, I planned to do both and got distracted by a tool call. The chat transcript does not distinguish these states reliably. The diff does. Treat the diff as ground truth and the chat as fiction; do not invert that ordering even when the chat is detailed and confident.

### 4.6 Output-shape verification

For any structured output an agent produces or consumes — tool inputs, structured returns, generated JSON, generated YAML, generated SQL — schema-validate before use. The 2026 default stack:

- **JSON Schema** for cross-language schemas.
- **Pydantic** (Python) for runtime validation and structured output.
- **Zod** (TypeScript) for runtime validation, increasingly the de facto standard for TS agents.

The discipline: every tool input and every structured output is schema-validated before consumption. On validation failure, retry with the validation error in the prompt — agents recover from schema errors quickly when the error message is specific.

### 4.7 Output-citation verification

A 2026-specific failure mode: the **convincing-but-fake citation**. The agent cites `src/foo.ts:142`. The file exists. Line 142 exists. The line at 142 has nothing to do with what was cited. The agent has hallucinated a coordinate that survives a casual file-exists check.

Defense: path-discipline tooling that mechanically resolves every `[link](path)` in agent output and checks (a) that the path exists, (b) that the line range cited contains content matching the claim's keywords. This is the mechanical equivalent of the diff-as-ground-truth habit, applied to references rather than changes.

### 4.8 Snapshot and golden testing for LLM outputs

The 2018-era `toMatchSnapshot()` does not survive contact with stochastic LLM outputs. A snapshot of a generation is updated every run; the test never fails and never passes meaningfully. (This is the snapshot-of-stochastic-output anti-pattern from §2.5.)

The 2026 successor is the **golden dataset with semantic-equivalence scoring**. A golden dataset is a curated, version-controlled collection of `(input, expected_output)` pairs. Actual outputs are compared against expected outputs using semantic-similarity scores rather than string equality.

Two configurations are in production use:

- **Tolerance bands.** Score actual against expected with a cheap embedding model; require cosine similarity ≥ 0.95 (or whatever threshold the dataset has been calibrated against). Fast, deterministic, weak signal.
- **LLM-as-judge.** A larger and slower model scores the actual output against a rubric (helpfulness, factuality, format adherence, hallucination presence). Slower, more expensive, much stronger signal. Increasingly the 2026 default for high-stakes paths.

The 2026 CI integration: a PR that touches prompts, embedding thresholds, retrieval configuration, or model versions triggers a Gold Set run; the LLM-as-judge scores the output against the rubric; the pipeline blocks the merge if similarity drops below the configured threshold or if hallucination rates spike. This is the LLM-output equivalent of the regression test suite.

### 4.9 Mutation testing as a spec-quality signal

The twocents.software 2026 post puts the case sharply: **"Coverage is a vanity metric on its own. 80% line coverage with weak assertions catches almost nothing… Claude Code and Cursor write plausible tests fast; they also write tests that pass without asserting anything meaningful."**

Mutation testing fixes this. The mutation-testing tool introduces small mutations into the implementation (negate a boolean, change `+` to `-`, swap `<` for `≤`) and checks whether the test suite catches the mutation. A surviving mutant means the test suite did not assert strongly enough on the mutated line. The mutation score is the fraction of mutants killed; it is a far better proxy for test-suite quality than line coverage.

2026 reference tools: **Stryker** (JavaScript, TypeScript, .NET, Scala), **Mutmut** (Python), **PIT** (Java). All three integrate with standard CI.

Recommended 2026 thresholds:

- **70% mutation score** on critical paths (auth, payment, data integrity).
- **50%** on standard features.
- **30%** on experimental or research code.

The agent-loop application is the highest-leverage use: when mutants survive, **feed the surviving mutants back to the agent** and ask it to strengthen the assertions that should have caught them. The agent is now generating against an adversarial signal (a mutated implementation it must distinguish from the real one) rather than against its own self-consistent fiction. This closes the loop the AI-reported-93-actually-34 gap was opened by.

## 5. Anti-patterns catalogue (2026)

A reference list for code review and CI policy. Each is a recurring failure observed in the field through 2026; each maps to a defense already named in this document.

- **"Agent says tests pass" without re-running.** Defense: §4.1 (test run this turn) and §4.4 (the claim table).
- **Tests that pass on first attempt.** The agent overfit the test to the output it just produced. Defense: §2.1 step 3 (watch the test fail) and §2.6 (property-based tests resist this).
- **Tests that skip side effects.** The function calls an API; the test mocks the call; the test passes; the API was never reached. Defense: §2.5 (mock-the-world smell) and integration tests against ephemeral environments (§4.3).
- **"I tested it manually."** Ad-hoc, not in CI, not reproducible, not in this turn. Defense: §4.1 (the test must run this turn, automatically).
- **Mock-the-world and tautological assertions.** §2.5.
- **Snapshot-of-stochastic-output.** §4.8 (use a golden dataset with semantic scoring instead).
- **"Coverage above 80%, ship it."** The AI-reported-93-actually-34 failure. Defense: §4.9 (mutation testing as the real coverage signal).
- **Verifying with the same model that generated.** Correlated failure modes — the same blind spot that produced the bug rates the output as clean. Defense: cross-family shadow comparison (a different model family scores the output; disagreement is signal, not noise).

## 6. Top 12 verification + TDD + spec patterns, ranked by leverage

Ranking is by *bug-prevention-per-unit-friction* as observed across 2025–2026 field reports. Lower-numbered patterns are usually cheaper to adopt and pay back faster.

1. **Run the verification command this turn.** The single highest-leverage habit; defeats the largest class of failure (agent narration mistaken for completion). Cost: near zero. (§4.1, §4.4)
2. **Diff before trust.** Run `git diff` after every "I did X" claim. Same near-zero cost; defeats the second-largest class of failure. (§4.5)
3. **Commit the failing test as a checkpoint.** Anthropic's non-obvious TDD instruction. Single highest-leverage TDD habit. (§2.2, §2.3)
4. **EARS-notation acceptance criteria.** Cheapest spec-side primitive; eliminates the largest class of misinterpreted-requirement bugs. (§3.4)
5. **Type-check on every change.** Catches the API-hallucination class of agent errors at near-zero per-call cost. (§4.1)
6. **One example test + one property test per behavior.** Combines documentation with adversarial coverage; the PBT papers (§2.6) show this combination dominating either alone.
7. **Schema-validate every structured output.** JSON Schema / Pydantic / Zod, retry on failure. (§4.6)
8. **Watch the test fail for the right reason before going green.** The superpowers TDD rule; prevents the most common test-quality bug. (§2.3)
9. **Mutation testing on critical paths.** The only reliable way to know what your tests are actually worth. (§4.9)
10. **Three-document spec before code.** `requirements.md` / `design.md` / `tasks.md` à la Kiro, or `/specify` / `/plan` / `/tasks` à la Spec-Kit. Higher upfront cost; pays back across the lifetime of the project. (§3.1, §3.2)
11. **Golden-dataset + LLM-as-judge in CI for LLM-using code.** The regression test suite for stochastic systems. (§4.8)
12. **Cross-family shadow verification.** A second model family scores the output; disagreement is the signal. Highest cost; reserved for highest-stakes paths. (§5)

The first five are free or near-free. Adopt them on day one. The middle four pay back within a sprint. The last three are investments — adopt them when the cost of an undetected failure exceeds the cost of the verification stack.
